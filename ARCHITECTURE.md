# 架构大白话

这份文档只讲 3 件事：

1. `Taotaotao` 在整个流程里扮演什么角色。
2. 新项目和治理工具仓之间怎么分工。
3. 为什么这套结构能让新的 IDE 里的 AI 快速进入工作状态。

## 1. `Taotaotao` 不是业务项目，它是本地工具仓

现在最准确的理解是：

- `Taotaotao` 负责保存规则、模板、项目类型和生成逻辑
- 新项目负责保存自己的本地事实、规则、代码和进展

也就是说，`Taotaotao` 更像一个本机常驻的“项目启动器”。

你真正开发业务时，主要工作目录应该是新项目目录，不是 `Taotaotao` 自己。

## 2. 这套结构吸收了最开始三篇文章的什么精神

核心不是“写很多 Prompt”，而是把 AI 协作流程固定下来。

最关键的 4 个点是：

- 先对齐，再动手
- 规则必须本地化，而不是靠聊天记忆
- 项目经验要能沉淀和回流
- 不同 IDE / agent 要吃同一套真源

所以这套工具真正做的是：

**把一套可复用的 AI 对话框架和工作框架，固化到本地文件。**

它不只是“让项目不依赖某个聊天会话”，更是让 AI 先受项目约束，再开始创作。

## 3. 当前的正确数据流

现在推荐的数据流是这样：

`Taotaotao 安装到本机`
-> `新 IDE 打开一个新项目目录`
-> `在新 IDE 终端里运行 bootstrap`
-> `当前目录生成 START_HERE + .agents + 业务骨架`
-> `AI 先读 START_HERE`
-> `AI 再读当前项目的 .agents 真源`
-> `如果需要，再回看本机 Taotaotao 文档`
-> `开始开发`

这里最重要的一点是：

AI 的第一阅读对象应该是“当前项目本地文件”，不是直接把整个治理仓当成当前项目。

## 4. 现在可以分成 4 层理解

### 第 1 层：本机工具层

也就是这个仓库本身：

- `vibe_governance/`
- `docs/`
- `references/`
- 根目录入口文档

这一层负责定义：

- CLI
- 规则目录
- 项目类型
- 模板

### 第 2 层：新项目入口层

`bootstrap` 后，新项目根目录会有：

- `START_HERE.md`
- `README.md`

这一层负责：

- 让人知道先看什么
- 让 AI 知道先读什么
- 限制 AI 不要上来就乱扫仓库

### 第 3 层：新项目真源层

也就是新项目里的 `.agents/`。

它负责保存：

- profile
- rules
- architecture decisions
- skills
- progress

这是 AI 真正该依赖的本地事实层。

### 第 4 层：适配输出层

包括：

- `AGENTS.md`
- `CLAUDE.md`
- `GEMINI.md`
- `.cursor/`
- `.github/`
- `.opencode/`

这一层只是把规则分发给不同工具，不是真源。

## 5. 为什么这套结构比“让 AI 自己扫仓库”更稳

因为它把 AI 的工作顺序固定了：

- 先读入口
- 再读真源
- 最后才看适配层

这样做的好处是：

- 不容易漏掉规则
- 不容易误把生成文件当真源
- 不容易把上一次会话残留记忆带进来
- 新 IDE、新账号、新 AI 更容易无缝接手

## 6. 当前版本已经做到什么

已经做到的：

- 可以把 `Taotaotao` 当成本地工具安装
- 可以在新 IDE 的当前目录直接 `bootstrap`
- 可以按项目类型生成治理骨架和业务目录骨架
- 可以生成本地 `skills`
- 可以生成 `START_HERE.md`
- 可以生成多 IDE / 多 agent 适配文件

还没做到的：

- 还没有细到按具体技术栈生成完整框架文件
- 还没有技术栈级脚手架模板体系
- 还没有自动把项目反馈提升成模板进化

所以当前版本已经是“本地项目启动器”，但还不是“全技术栈脚手架平台”。
