# 架构大白话

这份文档只讲最核心的 4 个问题:

1. 这个仓库为什么存在
2. 它现在是什么
3. 它现在不是什么
4. 它为什么要分层

## 一、它为什么存在

这个仓库的目标不是“写更多 Prompt”, 而是把 AI 协作流程固化到本地文件里。

它要解决的是:

- 换账号后上下文丢失
- 换 IDE 后规则不一致
- 多个 agent 各写各的流程
- 项目经验只留在聊天里

## 二、它现在是什么

它现在是:

- 一个治理内核原型
- 一个自举示例仓

也就是说, 它已经有:

- CLI
- `.agents/` 真源
- 规则目录和模板目录
- 多 IDE 适配层生成
- `PROGRESS` 记忆闭环

## 三、它现在不是什么

它现在还不是你最终想要的那种:

- 一条命令生成完整业务目录骨架
- 按项目类型自动生成 `embedded / backend / frontend / fullstack` 模板
- 自动把项目反馈升级成模板能力

这些是下一阶段的目标, 不是当前版本已经完成的能力。

## 四、为什么要分成 4 层

### 第 1 层: 根目录入口层

作用:

- 让人先看懂项目
- 给新账号和新 AI 一个稳定阅读顺序

### 第 2 层: `.agents/` 真源层

作用:

- 保存项目事实
- 保存项目规则
- 保存架构决策
- 保存经验记录

### 第 3 层: `vibe_governance/` 生成层

作用:

- 读取真源
- 读取 canonical 规则和模板
- 生成受管输出
- 做校验和同步

### 第 4 层: IDE / agent 适配层

例如:

- `AGENTS.md`
- `CLAUDE.md`
- `GEMINI.md`
- `.cursor/`
- `.github/`

作用:

- 把同一套规则发给不同工具
- 不让不同工具各自发明一套新流程

## 五、你现在应该怎么理解它

你可以把它理解成:

- 它已经把“治理地基”搭出来了
- 但还没有把“项目模板层”做出来

所以你现在看到的复杂感, 很大一部分来自于:

- 它既是程序源码仓
- 又是它自己的示例仓
