# 嵌入式项目专属使用指南

这份文档专门回答一个问题:

如果你要把这套治理体系用到 ESP32、STM32 这类软硬件协同项目里, 当前 v1 到底能做什么, 不能做什么, 应该怎么接。

这部分一定要先和 [SOURCE_MATERIALS.md](./SOURCE_MATERIALS.md) 一起看。因为最初原文里明确把嵌入式 / 硬件协同当成独立难点提出了, 但当前仓库没有假装自己已经把这件事做完。

## 一、先把当前边界讲透

当前 v1 已经为嵌入式项目做好的, 主要是三件事:

1. 给嵌入式项目提供治理骨架
2. 给跨账号接手提供本地记忆和版本追溯
3. 给 MCP / 硬件工具扩展预留 schema 和规则入口

当前 v1 还没有做好的, 也必须直接说清楚:

- 还没有完整的嵌入式自动化工作流
- 还没有自动生成的烧录、串口、联调规则
- 还没有完整的硬件 red line 规则包
- 还没有可直接拿来用的 MCP 工具契约

所以正确理解是:

- 它已经能帮你把嵌入式项目的“治理框架”搭起来
- 但还不能直接替你完成“硬件自动化工作流”

## 二、为什么嵌入式项目更需要这套治理骨架

纯软件项目接手时, 丢上下文已经很痛苦。

嵌入式项目更糟, 因为你丢的不只是代码背景, 还可能包括:

- 板卡型号
- 接线方式
- 供电条件
- 串口参数
- 烧录方式
- 固件与上位机协议边界
- bring-up 过程中的关键坑

这些信息如果只留在聊天里, 新账号基本不可能稳定接手。

所以对嵌入式项目来说, 这套仓库的价值首先是:

- 把项目事实写进 `.agents/`
- 把硬件红线写进规则和文档
- 把 bring-up 和联调经验写进 `PROGRESS`

## 三、当前最推荐的嵌入式接入路径

### 第 1 步: 初始化治理骨架

在嵌入式项目根目录运行:

```bash
python -m vibe_governance init --target . --project-type embedded
```

### 第 2 步: 先填项目事实, 不要先填技巧

优先修改:

- [`.agents/profile.yaml`](../.agents/profile.yaml)

建议至少先定清楚:

- `project_type`
- `project_lang`
- `comment_lang`
- `doc_mode`
- `agent_adapters`
- `mcp.enabled`
- `mcp.config_path`
- `mcp.allowed_tools`

如果这是一个真正的嵌入式项目, `project_type` 建议明确写成诸如 `embedded` 这样的值, 方便后续 overlay 演进时有稳定落点。

当前治理仓已经具备的最小落地能力包括:

- `init --project-type embedded` 生成默认治理骨架
- `project_version` 作为机器真源版本号
- `docs/DELTA_DECISIONS.md` 作为唯一增量真源模板
- `docs/NEXT_ITERATION_BASELINE.md` 作为下一轮纯索引模板
- `docs/VALIDATION_PLAN.md` 的 `M0 / M1 / M2 / M3` 模板骨架
- `validate` 对嵌入式最小文档和版本一致性做检查

### 第 3 步: 把硬件红线先写进 `.agents/RULES.md`

最先该写的不是“优化建议”, 而是红线。

例如:

- 哪些引脚绝对不能乱改
- 哪些供电条件不能违反
- 哪些烧录动作必须人工确认
- 哪些测试步骤必须先断电 / 先复位 / 先拔外设
- 哪些设备操作不能交给 agent 自动执行

当前 v1 还不会把这些全部自动变成机器强制规则, 但它至少能保证这些内容留在仓库里, 后续人和 AI 都能读到。

### 第 4 步: 把系统边界写进架构决策

去改:

- [`.agents/architecture-decisions.yaml`](../.agents/architecture-decisions.yaml)

至少建议固化这些内容:

- 主控型号
- 工具链
- 烧录方式
- 调试方式
- 固件与上位机边界
- 上位机与前端边界
- 协议层边界

### 第 5 步: 用 `PROGRESS` 记录 bring-up 和联调

嵌入式项目比纯软件更需要结构化经验记录。

建议优先写进 `.agents/progress/entries/*` 的内容有:

- 首次点灯 / 首次联网 / 首次上报成功
- 串口参数踩坑
- 供电问题
- 工具链版本问题
- 烧录器或驱动问题
- 固件与上位机协议调整
- 传感器接线或时序问题

## 四、当前 `mcp` 字段到底有什么用

`.agents/profile.yaml` 现在已经预留了:

- `mcp.enabled`
- `mcp.config_path`
- `mcp.allowed_tools`

当前这些字段的真实作用是:

- 给项目保存“是否准备引入 MCP”的事实
- 给后续扩展留标准入口
- 在开启时配合占位规则提醒“当前只是预留, 不是完整契约”

当前它们不代表:

- 只要开启就能自动联动硬件
- 只要填配置就能自动生成烧录脚本
- 只要列了工具就能安全地让 agent 直接操作设备

## 五、嵌入式项目的文档和记忆应该怎么落点

### 适合写在根目录说明文档里的

- 项目总体结构
- 固件 / 上位机 / 前端的关系
- 项目如何接手
- 版本迭代说明

### 适合写在 `.agents/RULES.md` 里的

- 项目特有红线
- 人工确认要求
- 设备操作禁令
- 高风险动作前置条件

### 适合写在 `.agents/architecture-decisions.yaml` 里的

- 系统边界
- 工具链选型
- 关键通信协议边界
- 硬件 / 软件 / 前端的职责拆分

### 适合写在 `PROGRESS` 里的

- bring-up 过程
- 联调过程
- 踩坑记录
- 候选回流经验

## 六、嵌入式项目的版本迭代应该额外记什么

嵌入式项目建议比纯软件更重视版本上下文。

每次进入版本级变化时, 最好明确记录:

- 固件版本改了什么
- 上位机是否跟着改
- 协议是否有不兼容变化
- 烧录方式是否变化
- 调试工具是否变化
- 对应的 `PROGRESS` 条目有哪些

推荐的最小闭环是:

1. 在 `.agents/progress/entries/*` 记录联调和踩坑
2. 在 `docs/DELTA_DECISIONS.md` 记录仍然生效的增量决议
3. 在 `docs/NEXT_ITERATION_BASELINE.md` 维护下一轮必读索引
4. 在 [CHANGELOG.md](../CHANGELOG.md) 记录版本级变化
5. 在 [ARCHITECTURE.md](../ARCHITECTURE.md) 或架构决策里写清边界变化
6. 在 [CONTEXT_MIGRATION.md](./CONTEXT_MIGRATION.md) 里补充新账号接手所需环境差异

## 六点五、如何判断是 L0 / L1 / L2

嵌入式三层项目建议按“是否影响跨层契约”来判断:

### L0 示例

- 固件内部重构, 但不改对外状态语义
- 网关内部容错实现优化, 但超时表现和错误语义对外不变

### L1 示例

- 某个实现层补丁会影响下一轮 AI 对行为的理解
- 但协议文档、schema、硬件清单里还没有明文要求

### L2 示例

- 硬件采样窗口变化, 且影响校准口径或验收标准
- 网关 ACK 超时从 3s 变到 5s, 且协议或对外状态机依赖该表现
- 前端消费的状态字段、错误码或阶段语义发生变化

有疑问时, 优先判为 `L2`。

## 七、跨账号同步时, 嵌入式项目要额外同步什么

除了通用治理文件, 嵌入式项目还应特别确认这些资料已经进仓库:

- 板卡型号说明
- 接线说明
- 烧录步骤
- 调试串口参数
- 本地工具链版本
- 协议文档
- 关键硬件红线

如果这些东西没写进仓库, 新账号就算拿到了代码, 也很难真正接手项目。

## 八、如果你要把嵌入式能力推进到下一阶段, 建议怎么做

不要直接在现有模板里塞大量条件分支。

更稳的顺序是:

1. 先在真实嵌入式项目里把通用红线沉淀成文档和 `PROGRESS`
2. 再从这些经验里提炼通用 `rule_id`
3. 再决定是否引入真正的 `embedded` overlay
4. 最后才补 MCP 工具契约和自动生成逻辑

## 九、环境兼容性规则

嵌入式项目经常涉及跨平台开发和多工具链协同, 以下是必须遵守的环境规则:

### PowerShell 中文输出

- Windows 环境下使用 PowerShell 时, 如果输出包含中文, 必须确保终端编码为 UTF-8
- 推荐在脚本开头添加: `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8`
- 或在 PowerShell 配置中设置: `$OutputEncoding = [System.Text.Encoding]::UTF8`

### UTF-8 文件编码

- 所有文本文件必须使用 UTF-8 编码保存
- 特别注意:
  - Python 源文件
  - Markdown 文档
  - YAML 配置文件
  - 协议文档
  - 测试计划
- 避免使用 UTF-8 BOM, 优先使用无 BOM 的 UTF-8

### pytest 临时文件清理

- pytest 运行后会在 `__pycache__` 和 `.pytest_cache` 目录生成临时文件
- 这些目录已在 `.gitignore` 中排除, 不应提交到版本库
- 如需清理, 运行: `find . -type d -name "__pycache__" -exec rm -rf {} +` (Linux/macOS) 或 `Get-ChildItem -Recurse -Directory -Filter "__pycache__" | Remove-Item -Recurse -Force` (PowerShell)

### 路径兼容性

- 代码中的路径分隔符使用 `os.path.join()` 或 `pathlib.Path` 处理
- 避免硬编码 `/` 或 `\`
- 脚本中的路径使用相对路径或通过配置文件指定

### 工具链版本

- 在 `docs/HARDWARE_BRINGUP.md` 中明确记录:
  - Python 版本要求
  - 编译器版本
  - 烧录工具版本
  - 串口工具版本
- 版本不一致可能导致编译失败或烧录异常

## 十、给嵌入式维护者的一句提醒

当前 v1 已经能帮你解决:

- 规则不落地
- 历史容易丢
- 换账号难接手
- 多 IDE 规则不一致

但它还没有替你解决:

- 真正的硬件自动化
- 真正的 MCP 硬件联动
- 完整的嵌入式安全规则包

所以最稳的用法就是:

- 先用它把治理骨架搭稳
- 再把嵌入式经验逐步沉淀成下一阶段能力
