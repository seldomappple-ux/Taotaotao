# Superpowers 多 Agent 协同编程分析

## 文档定位

这份文档只分析 `obra/superpowers` 对多 agent 共同编程的启发, 不混入其他项目.

分析视角是架构设计视角, 重点回答 5 个问题:

1. 它如何组织多 agent 的工作方式.
2. 它如何控制上下文膨胀.
3. 它如何做并行与隔离.
4. 它如何把结果拉回主线并校验.
5. 哪些做法适合 Taotaotao 借鉴, 哪些不适合直接照搬.

## 一, 项目本质定位

`superpowers` 本质上不是一个"多 agent 调度平台", 而是一套面向 AI 编码终端的增强工作流库.

它的核心抽象不是"中心控制器", 而是:

- `skills`: 把高频工程动作封装成可复用能力单元
- `commands`: 把常见任务入口标准化
- `hooks`: 把关键时机接入自动化动作
- `plugins`: 适配不同 coding agent 终端

从架构角度看, 它做的事情非常清晰:

- 不试图替代 agent 自身
- 不强行引入一个庞大的调度层
- 直接围绕"怎样让多个 agent 更稳定地协同工作"提供操作规范

这意味着它的设计哲学偏"轻编排, 重工作习惯标准化".

## 二, 多 Agent 协作的核心架构

### 1. 以 Skill 代替硬编码流程

`superpowers` 的第一层设计不是把流程写死在程序里, 而是把经验固化成 skill 文档.

这带来 3 个结果:

- agent 在接到任务后, 有可复用的最佳实践入口
- 多个 agent 做事时, 容易遵循同一套路
- 工作流能快速演化, 不必先修改底层系统

对多 agent 协作来说, 这是一种很实用的设计. 因为多 agent 真正最容易失控的地方, 往往不是缺少"再来一个 agent", 而是不同 agent 各用各的方法.

### 2. 以任务拆分替代上下文硬撑

`subagent-driven-development` 和 `dispatching-parallel-agents` 两个 skill 传达了一个很强的架构观点:

- 主 agent 不应该自己吞下全部上下文
- 主 agent 要先做任务拆解
- 拆成若干个上下文清晰的小任务后, 再交给子 agent 执行

这里的关键不是"并行"本身, 而是"可独立交付的任务单元".

也就是说, `superpowers` 把多 agent 协作建立在以下链路上:

`任务分解 -> 子任务边界清晰 -> 并行派发 -> 各自产出 -> 主线整合`

这比"一上来同时开很多 agent"成熟得多.

### 3. 以 Worktree 隔离替代共享工作区争抢

`using-git-worktrees` 对多 agent 协作非常关键.

它强调:

- 不同任务最好在独立 worktree 中执行
- 每个 agent 尽量拥有自己的隔离工作目录
- 减少同一工作区并发编辑带来的冲突

从架构师视角看, 这是 `superpowers` 最有价值的部分之一. 因为多 agent 最大的工程问题之一不是不会写, 而是互相覆盖.

如果没有隔离层, 常见问题会马上出现:

- 文件相互覆盖
- git 状态变脏且难追踪
- 回滚边界不清晰
- 集成时难以判断是谁引入的问题

而 worktree 模式天然提供了:

- 任务级隔离
- 分支级隔离
- 结果回收前的独立验证空间

这非常适合需要多人/多 agent 并行推进的工程环境.

## 三, 它如何解决多 Agent 共同编程的关键问题

### 1. 角色问题: 用"主 agent + 子 agent"而不是对等群聊

`superpowers` 倾向于采用分层协作:

- 主 agent: 理解目标, 拆任务, 决定派发策略, 负责汇总
- 子 agent: 处理范围明确的局部任务

这说明它并不鼓励所有 agent 对等讨论后自由修改代码, 而是采用更工程化的主从式结构.

优点是:

- 决策链更短
- 汇总责任明确
- 不容易发生任务重复和边界漂移

缺点是:

- 主 agent 的拆分质量会成为上限
- 如果主 agent 没拆好, 子 agent 会一起偏航

### 2. 上下文问题: 用局部任务包控制 prompt 体积

`dispatching-parallel-agents` 的思路不是复制全部上下文给每个 agent, 而是尽量只下发完成该任务所需的信息.

这背后对应的是经典架构原则:

- 让每个执行单元只承担必要上下文
- 把共享背景压缩成稳定约束
- 把局部任务说明写得足够精确

这能直接降低:

- token 浪费
- 指令漂移
- 多 agent 因理解不同而产出不一致

### 3. 并行问题: 并行不是目的, 吞吐量才是目的

`superpowers` 对并行的处理很克制.

它不是鼓励"能并行就并行", 而是强调:

- 找出彼此独立的任务
- 在不会相互阻塞时再并行
- 并行后要有明确回收与 review 机制

这说明它把并行视为吞吐量优化手段, 而不是工作流主角.

这是成熟系统的表现. 因为大量团队在多 agent 试验里失败, 都是因为把并行当成了能力本身, 忽略了整合成本.

### 4. 质量问题: 用 Review 作为回流关口

`requesting-code-review` 提供的不是普通的"看一下代码", 而是把 review 视作主线合流前的强制关口.

从多 agent 协作看, 它至少解决了 3 个问题:

- 子 agent 的产出需要二次核验
- 主 agent 不应该无条件信任局部实现
- 合并动作前应先识别风险点和缺失测试

这代表 `superpowers` 的一个重要观点:

多 agent 提高的是吞吐量, 但如果没有 review 机制, 缺陷也会被高速放大.

### 5. 规划问题: 用 Plan 先锁边界, 再开工

`writing-plans` 和子 agent skill 共同体现出一个顺序:

1. 先写计划
2. 明确任务边界
3. 再派发给子 agent
4. 最后回收与审查

这说明它并不是一个即时 improvisation 风格的系统, 而是明显偏向"先收敛, 后执行".

对复杂项目来说, 这很重要. 因为多 agent 在没有计划约束时, 最容易出现以下问题:

- 重复造轮子
- 文件命名不一致
- 改动边界失控
- 测试和实现脱节

## 四, 从架构师角度看它的强项

### 1. 强在"组织 agent 行为", 不强在"炫技调度"

它最成熟的地方不是某个复杂框架, 而是把多 agent 协作最容易翻车的环节做成了稳定动作:

- 先拆分
- 再隔离
- 再并行
- 再 review

这套动作链对工程成功率的帮助, 远大于单纯增加 agent 数量.

### 2. 强在上下文治理

很多多 agent 方案把重点放在怎么把更多信息喂给更多模型. `superpowers` 反过来做:

- 尽量缩小每个 agent 的局部上下文
- 用清晰边界替代全量共享
- 用技能模板保证执行方式一致

这非常符合真实工程环境.

### 3. 强在兼容不同 agent 终端

它通过 plugin 机制去适配不同 coding agent.

这说明其设计目标不是绑定单一供应商, 而是维持跨终端的一致协作习惯. 对长期团队治理来说, 这比单点功能更有价值.

### 4. 强在接近一线开发者实际使用方式

它没有强制要求企业级平台、集中任务总线或复杂后端.

所以它适合:

- 本地开发
- 小团队协作
- 多终端混合使用
- 个人开发者逐步升级工作流

## 五, 它的局限与边界

### 1. 它更像方法库, 不是完整编排平台

如果团队期待的是:

- 自动任务分发中心
- 全局依赖图调度
- 可视化任务状态面板
- 自动冲突仲裁

那么 `superpowers` 不是直接答案.

它提供的是高质量作业方法, 不是完整 orchestration platform.

### 2. 它依赖使用者具备较强任务拆分能力

它的很多收益来自主 agent 或操作者能不能把问题拆对.

一旦拆分不合理, 即便 skill 模板存在, 效果也会明显下降.

### 3. 它更适合中小规模并行, 不一定适合超复杂组织级协同

对于跨团队、跨仓库、长周期、多阶段审批的大型系统, 只靠 skill + worktree 还不够.

通常还需要补:

- 任务状态系统
- 结构化依赖管理
- 统一验收标准
- 更强的制品回溯链路

### 4. 它默认开发者愿意遵循纪律

`superpowers` 的成功前提是:

- 愿意先计划
- 愿意分任务
- 愿意隔离工作区
- 愿意做 review

如果使用者习惯直接在主分支里边想边改, 那它的价值会大打折扣.

## 六, 对 Taotaotao 的可借鉴点

结合当前 Taotaotao 的治理现状, 我认为最值得吸收的是下面 6 点.

### 1. 建立"主 agent 负责拆分, 子 agent 负责局部交付"的稳定模式

Taotaotao 目前已经有治理文档基础, 但在后续升级中, 还需要更明确地区分:

- 谁负责问题归纳
- 谁负责局部验证
- 谁负责最终整合

可以把这套分工沉淀为固定规则, 避免每次升级都重新摸索.

### 2. 把"先计划再派发"写成硬规则

对于软硬件协同项目, 先计划尤为重要.

建议把以下内容写进升级前置检查:

- 本轮目标
- 软硬件边界
- 任务拆分清单
- 哪些可以并行
- 哪些必须串行
- 验收口径

### 3. 引入任务级隔离工作区

如果未来 Taotaotao 真要进入多 agent 并行开发阶段, 建议认真引入 worktree 或等价隔离机制.

尤其是以下场景:

- 上位机和下位机并行改动
- 协议层和 UI 层并行改动
- 文档治理和代码实现并行推进

这能显著减少互相覆盖.

### 4. 把 Review 从建议动作升级为必经关口

当前项目在治理层已经强调沉淀和校验, 下一步建议进一步明确:

- 子任务产出不能直接并入主线
- 先经过 review 或 checklist
- 再进入 summary / plan / implementation

这对多 agent 协作尤其关键.

### 5. 把高频经验封装成项目内 skill 或模板

Taotaotao 已经开始沉淀治理经验, 下一步可以考虑把高频动作模板化, 比如:

- 升级问题归纳模板
- 嵌入式串口/网络联调检查模板
- 文档写入与 UTF-8 校验模板
- 上下位机协议验证模板

这和 `superpowers` 的 skill 思路高度一致, 但可以更贴合本项目.

### 6. 把并行作为手段, 不是目标

Taotaotao 当前更需要的是稳定升级, 不是盲目堆多 agent.

因此推荐策略是:

- 先把单 agent 工作流固定
- 再挑出真正独立的任务做并行
- 最后再逐步增加 agent 数量

## 七, 不建议直接照搬的点

### 1. 不建议照搬其通用 skill 颗粒度

Taotaotao 是带明显嵌入式和软硬件协同特征的项目.

如果直接照搬通用 software skill, 很容易漏掉:

- 硬件初始化前置条件
- 通信链路确认
- 真实设备握手检查
- Mock 与真机切换规则

因此 Taotaotao 需要的是"领域化 skill", 不是通用 skill 原样复制.

### 2. 不建议只做 worktree, 不补文档治理

`superpowers` 的重点更多在执行层方法. 但 Taotaotao 当前暴露出的一个真实问题, 是文档层次和根目录结构也需要治理.

所以不能只吸收其并行开发经验, 还要同步做好:

- 目录分层
- 文档分工
- 版本化升级记录
- 原始记录与总结分离

### 3. 不建议过早追求过多并行

当前 Taotaotao 还处在治理结构收敛阶段.

如果现在就激进上多 agent 并发实现, 风险会高于收益. 更稳妥的顺序应该是:

1. 先定文档和规则
2. 再定版本边界
3. 再定任务拆分
4. 最后才是并行执行

## 八, 结论

`superpowers` 给多 agent 共同编程带来的最大启发, 不是"怎么同时叫更多 agent 干活", 而是"怎样让多个 agent 在工程纪律下协同".

它真正有价值的架构思想可以概括成一句话:

`先收敛边界, 再拆分任务, 再隔离执行, 最后审查回流.`

如果 Taotaotao 后续要持续升级成适合软硬件协同的多 agent 治理项目, 那么最值得吸收的不是它的表面形式, 而是这条稳定主线:

- 统一工作方法
- 限制上下文膨胀
- 隔离并行执行空间
- 强制 review 后合流

这条主线非常值得纳入 Taotaotao 后续版本治理设计中.
