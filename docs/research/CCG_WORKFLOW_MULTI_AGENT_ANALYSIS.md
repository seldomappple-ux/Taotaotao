# CCG Workflow 多 Agent 协同编程分析

## 文档定位

这份文档只分析 `fengshao1227/ccg-workflow` 对多 agent 共同编程的启发, 不混入其他项目.

分析重点聚焦 5 个问题:

1. 它如何组织多模型 / 多 agent 的分工.
2. 它如何把协作流程固定成稳定流水线.
3. 它如何控制上下文、权限和并行风险.
4. 它如何把局部产出拉回统一主线.
5. 哪些做法值得 Taotaotao 借鉴, 哪些不宜直接照搬.

## 读取范围

本次分析主要基于以下公开材料:

- `README.zh-CN.md`
- `CLAUDE.md`
- `templates/commands/workflow.md`
- `templates/commands/team-research.md`
- `templates/commands/team-plan.md`
- `templates/commands/team-exec.md`
- `templates/commands/team-review.md`

项目地址:

- <https://github.com/fengshao1227/ccg-workflow>

## 一, 项目本质定位

`ccg-workflow` 本质上不是单纯的提示词集合, 而是一套面向 Claude Code 的多模型协作编排系统.

它的核心设计非常鲜明:

- Claude 负责编排、决策、审核
- Codex 主攻后端、逻辑、调试
- Gemini 主攻前端、UI、交互

从架构上看, 它不是让多个模型自由对话, 而是先定义角色分工, 再把工作流做成命令系统.

这意味着它的设计哲学偏:

- 强编排
- 强流程
- 强角色路由
- 强审查门控

如果说 `superpowers` 更像"多 agent 方法库", 那 `ccg-workflow` 更像"多 agent 执行中台的轻量实现".

## 二, 多 Agent 协作的核心架构

### 1. 中心编排者明确存在

`ccg-workflow` 的第一原则不是模型平权, 而是中心编排.

它把 Claude 放在中心位置, 负责:

- 选择命令入口
- 收敛需求
- 决定执行节奏
- 综合 Codex 和 Gemini 的输出
- 在最终写入前审查 Patch

这是一种非常典型的 orchestrator 架构.

它的本质不是"三个模型一起干活", 而是:

`Claude 编排 -> 外部模型分工产出 -> Claude 统一收口`

这个收口动作非常关键. 没有它, 多模型协作很容易变成三份风格不一致的局部答案.

### 2. 角色分工不是建议, 而是系统默认

从 README 和 `CLAUDE.md` 可以看出, 该项目把模型路由明确固化为默认规则:

- 前端任务优先走 Gemini
- 后端任务优先走 Codex
- Claude 负责 orchestration 和 review

这说明它采用的不是通用 agent 池, 而是"按擅长领域分配职责".

这种设计的价值在于:

- 降低每次手动选模型的成本
- 避免同一任务在多个模型之间反复摇摆
- 让输出风格相对稳定

但同时也意味着, 系统效果高度依赖这套分工是否真的适配项目类型.

### 3. 工作流被做成命令化流水线

`workflow.md` 明确了完整顺序:

`研究 -> 构思 -> 计划 -> 执行 -> 优化 -> 评审`

每一阶段都有几个特点:

- 阶段职责单一
- 阶段之间要停下来确认
- 低评分或未批准就停止
- 后续阶段尽量复用前阶段产出

这代表 `ccg-workflow` 的核心思想不是让 agent 连续自由发挥, 而是让协作被阶段门控.

从工程治理角度看, 这种分段比"一条长 prompt 从头写到尾"可靠得多.

## 三, 它如何解决多 Agent 协同中的关键问题

### 1. 用 Prompt 增强先收敛需求

无论是 `workflow` 还是 `team-research`, 它都把 Prompt 增强放在最前面, 甚至是 mandatory.

这背后反映出一个很专业的判断:

多 agent 系统最怕的不是模型不够多, 而是需求输入本身模糊.

所以它在派发前先做:

- 目标补全
- 缺失信息识别
- 边界补全
- 验收标准补全

这样做的收益很大:

- 后续 agent 接到的是结构化任务
- 并行分析时不容易各说各话
- review 时能有明确的判断基准

### 2. 用约束集替代开放式 brainstorming

`team-research` 的产物被定义为"约束集", 而不是泛泛的信息堆砌.

这是一条非常成熟的架构思路.

它的意思是:

- 研究阶段先识别哪些方向不能走
- 把技术边界、现有模式、依赖、风险先锁住
- 后续 plan 阶段尽量做到零决策

这对多 agent 特别重要. 因为一旦约束不清, 多个 agent 并行后很容易输出互相冲突的方案.

### 3. 用文件化状态传递替代长会话堆叠

README 明确提到 Team 系列工作流每步都可以 `/clear`, 再通过文件传递状态.

例如:

- 研究结果写到 `.claude/team-plan/...-research.md`
- 计划阶段读取研究结果
- 执行阶段读取计划文件
- 审查阶段再读取计划与 diff

这代表它对上下文问题的回答不是"继续保持聊天上下文", 而是:

- 把关键状态外化为文件
- 让上下文能被清空
- 让阶段状态可追溯、可重启、可恢复

这是它对多 agent 协作最有价值的设计之一.

### 4. 用 Patch 权限模型控制外部 agent 风险

README 中一个非常强的设计点是:

- 外部模型没有直接写权限
- 外部模型只返回 Patch
- 由 Claude 审核后应用

从系统安全性来看, 这是一个非常合理的权限边界:

- 规划和实现能力可以外包
- 最终落盘权留在中心编排者
- 外部模型不直接污染主工作区

这解决了多 agent 共同编程里一个很现实的问题:

不是所有 agent 都应该拥有同级写权限.

### 5. 用 Team 命令显式管理并行执行

`team-plan`、`team-exec`、`team-review` 共同组成了一条面向并行开发的专用链路.

其中最值得注意的是 `team-exec`:

- 先解析计划中的子任务、依赖关系、并行分组
- 再按 Layer 分组并行派发
- Builder teammate 只负责各自子任务
- 每个 Builder 都有硬性文件范围约束

这说明它不是简单地"多开几个 agent", 而是在做受约束的并行执行.

这里的关键有 3 个:

- 并行要按依赖层分批次
- 任务边界要细到文件范围
- 执行者必须遵守写入边界

这一套是非常工程化的.

### 6. 用双模型交叉审查降低单点盲区

`team-review` 的核心不是普通 review, 而是 Codex 和 Gemini 双模型交叉审查.

并且审查结果会被分成:

- `Critical`
- `Warning`
- `Info`

然后再通过门控规则决定:

- `Critical > 0` 必须修复或经用户确认跳过
- `Critical = 0` 才能通过

这意味着 `ccg-workflow` 把 review 设计成了一个真正的质量关卡, 而不是可有可无的建议步骤.

## 四, 从架构师角度看它的强项

### 1. 强在把多 agent 协作做成可执行流程

很多多 agent 项目停留在理念层: 说要分工、说要并行、说要审查.

`ccg-workflow` 的优势是把这些理念都落成了命令和阶段:

- 有入口命令
- 有阶段产物
- 有文件落点
- 有审查门槛

这让协作不再依赖个人记忆.

### 2. 强在把角色差异显式化

它不回避模型差异, 反而主动利用差异:

- Codex 偏逻辑与后端
- Gemini 偏前端与体验
- Claude 负责整合与守门

这比"所有模型都做所有事"更节省认知成本.

### 3. 强在上下文治理和恢复能力

通过 `/clear` + 文件化状态传递, 它实际上建立了阶段型检查点.

这对复杂项目很有帮助, 因为:

- 中断后容易恢复
- 上下文过长时不会整体崩掉
- 新 agent 可以从文件继续接手

### 4. 强在权限与风险控制

Patch 审核模型是它非常成熟的一点.

对于多模型系统来说, 直接给每个 agent 写权限虽然省事, 但风险极高. `ccg-workflow` 选择了更稳的路线.

### 5. 强在适合命令化重复执行

它把常用协作动作收敛成大量 `/ccg:*` 命令.

这意味着:

- 团队成员学习成本更低
- 使用动作更一致
- 治理经验更容易标准化复制

## 五, 它的局限与边界

### 1. 它强依赖 Claude 作为中心控制器

这种设计的最大优点是统一, 最大限制也是统一.

一旦中心编排者质量下降, 整体系统的拆分、整合、审核质量都会一起下降.

### 2. 它的角色分工适合 Web 开发, 不一定天然适合嵌入式

该项目强调前端与后端分治, 这对于 Web 工程非常自然.

但 Taotaotao 当前更偏:

- 上位机
- 下位机
- 协议层
- 驱动层
- 联调与握手

这种结构不完全等于前后端结构, 所以不能机械照抄 Codex/Gemini 分工方式.

### 3. 它的命令体系较重

27 个命令带来的收益是全面, 代价是:

- 上手曲线增加
- 命令之间边界需要长期维护
- 规范漂移时要同步更新大量模板

对 still evolving 的项目来说, 太早堆满命令系统可能反而增加维护成本.

### 4. 它对执行环境有较多前置依赖

例如:

- Claude Code
- Codex CLI
- Gemini CLI
- jq
- 特定环境变量
- Agent Teams 实验特性

这意味着它更适合已经接受这套工具链的团队, 不一定适合最轻量的本地治理场景.

### 5. 它依赖高质量计划文件

`team-exec` 和 `team-review` 的质量, 本质上都建立在研究结果和计划质量之上.

如果计划文件对子任务边界、文件范围、依赖层级写得不准确, 并行执行就会变得危险.

## 六, 对 Taotaotao 的可借鉴点

结合当前项目现状, 我认为最值得吸收的是下面 7 点.

### 1. 引入中心编排角色, 但保留领域化分工

Taotaotao 后续如果要做多 agent 协作, 很适合借鉴"中心编排者"这个概念.

但角色划分不应照抄前端 / 后端, 更适合改成:

- 架构编排 agent
- 下位机 / 驱动 agent
- 上位机 / 工具链 agent
- 文档与治理 agent

这样更贴合本项目.

### 2. 把升级流程做成显式阶段

当前项目已经有升级原始记录、升级总结、版本 summary, 下一步可以继续借鉴 `ccg-workflow` 的阶段思想:

- 研究阶段: 原始问题归档与补全
- 约束阶段: 找出硬约束与现有边界
- 计划阶段: 形成 vX.Y.Z 实施计划
- 执行阶段: 分任务落地
- 审查阶段: 回看 checklist 与风险

### 3. 学它的"文件化状态传递"

Taotaotao 已经开始做文档沉淀, 这很适合继续强化为:

- 原始记录文件
- 约束与分析文件
- 计划文件
- 执行记录文件
- 审查文件

这样以后即使切换 IDE、切换 agent、切换会话, 都能稳定恢复.

### 4. 学它的权限边界思想

对于未来的多 agent 实现, 最好不要一开始就让所有 agent 都直接修改主工作区.

更稳妥的做法是:

- 局部 agent 输出 patch / 建议
- 主 agent 审核
- 通过后再合入

这对软硬件项目尤其重要, 因为错误修改的代价通常比普通文档项目更高.

### 5. 学它的并行前置条件

`ccg-workflow` 很强调:

- 先有研究
- 再有计划
- 再有并行执行

这个顺序非常适合 Taotaotao 当前阶段.

现在我们最不该做的, 就是还没把结构理清就盲目多 agent 并发修改.

### 6. 学它的文件范围约束

对 Taotaotao 未来并行改造特别有用的一点是:

- 每个子任务都明确到文件范围
- 超出范围视为越界

这个机制很适合解决你之前反复提到的结构混乱和文件命名失控问题.

### 7. 学它的分级审查

后续可以把升级检查也分成:

- `Critical`: 必须先修
- `Warning`: 建议本轮修
- `Info`: 作为后续优化

这样总结文档会更利于执行, 不会所有问题都堆在一起.

## 七, 不建议直接照搬的点

### 1. 不建议直接照搬前端 / 后端二元分工

Taotaotao 的真实问题空间不是纯 Web.

如果直接套"Gemini 负责前端, Codex 负责后端"这套说法, 容易掩盖真正需要拆开的边界:

- 设备侧
- 主机侧
- 协议侧
- 文档治理侧

### 2. 不建议一开始就引入过重命令体系

当前 Taotaotao 还在治理收敛期.

如果现在就一次性引入大量命令, 可能会出现:

- 文档变多但不好用
- 入口增多但执行更乱
- 用户自己也记不住流程

更稳妥的方式是先保留少数关键命令或关键模板.

### 3. 不建议把所有外部 agent 都挂在同一工具链前提上

`ccg-workflow` 对 Claude Code 生态集成很深.

但 Taotaotao 未来可能需要兼容多 IDE、多代理环境, 所以要吸收其思想, 不要过度绑定其具体实现.

### 4. 不建议过早依赖实验性团队特性

Agent Teams 很强, 但也意味着:

- 环境要求更高
- 行为稳定性更依赖宿主平台
- 排错成本更高

对当前阶段来说, 更建议先把文档驱动 + 任务驱动稳定下来.

## 八, 结论

`ccg-workflow` 对多 agent 共同编程的最大启发, 不在于它用了多少模型, 而在于它把多模型协作变成了一条可执行、可中断、可恢复、可审查的命令化流水线.

它最值得吸收的主线可以概括为:

`先增强需求 -> 再提炼约束 -> 再生成计划 -> 再按边界并行 -> 最后统一审查与收口`

对 Taotaotao 来说, 这份经验最有价值的地方不是"照搬工具", 而是学习它这几条工程原则:

- 需要中心编排者
- 需要阶段化检查点
- 需要文件化状态传递
- 需要权限边界
- 需要分级审查

如果后续我们真的要把 Taotaotao 推进到多 agent 协同升级, `ccg-workflow` 提供的是一套非常值得参考的流程骨架, 但必须经过领域化改造后再落地.
