# v1.0.0 实施计划

这份文档是 `v1.0.0` 的执行计划正文, 只负责说明怎么实施, 不承担问题归纳和计划评价职责。

相关文档分工如下:

- `v1.0.0-summary.md`: 问题总结与版本总览
- `v1.0.0-plan.md`: 实施计划正文

## 一、版本定位

`v1.0.0` 的收口线固定为:

- `P0` 全量落地
- `P1` 只做最小护栏

本版目标不是一次做成完整嵌入式平台, 也不是直接做成多 agent 运行平台, 而是把当前仓库升级为“可正式承接嵌入式治理项目”的第一版。

## 二、成功标准

本版成功以以下 4 条为准:

1. 能通过 `init --project-type embedded` 生成一套可用的嵌入式治理骨架。
2. 能在项目启动前明确软硬件链路、通信模式、关键硬件前提和验证路径。
3. 能通过 `validate` 做机器校验, 且版本号默认不自动升版。
4. 能把根目录入口、升级文档、命名规则和提交消息规则收敛到可持续维护状态。

## 三、关键接口

### 1. `init --project-type embedded`

- `vibe-governance init` 新增 `--project-type`
- 支持值:
  - `governance`
  - `embedded`
- 默认值:
  - `governance`

### 2. `.agents/profile.yaml` 的 `project_version`

- 新增必填字段: `project_version`
- 机器真源格式固定为 `major.minor.patch`
- 示例: `1.0.0`
- 不使用 `v1.0.0` 作为机器真源
- 本版不支持 `-alpha`、`-rc` 等后缀
- 展示层文档可写 `v1.0.0`, 但校验和生成统一读取 `project_version`

### 3. `validate` 扩展范围

`validate` 在本版继续沿用原命令名, 不新增 `doctor`, 但要扩展为嵌入式项目校验入口, 至少覆盖:

- 嵌入式骨架必需文档存在性
- 嵌入式关键字段存在性
- 版本号格式与版本一致性
- 协议 / Schema 示例 / 测试计划联动检查
- 中文 UTF-8 护栏检查

### 4. 默认目录规则 + 可覆盖策略

- 本版提供默认嵌入式目录骨架
- 默认可以采用:
  - `.c` 进 `src/`
  - `.h` 进 `lib/`
  - 新模块默认成对生成 `.c + .h`
- 这只是默认值, 不作为唯一结构
- 允许通过 profile 或等价配置覆盖目录约定, 以适配不同嵌入式项目

## 四、P1 最小护栏范围

为了防止 `P1` 范围失控, 本版明确只做到下面程度:

### P-004 版本同步

- 做:
  - 统一版本真源
  - 统一版本引用
  - 版本一致性检查
- 不做:
  - 版本依赖图
  - 自动版本 bump

### P-006 环境治理

- 做:
  - 规则文档
  - `validate` 最小护栏
- 不做:
  - Git Hook 模板
  - Git Hook 自动安装

### P-009 根目录治理

- 做:
  - 目录说明
  - 阅读顺序
  - 入口收敛
- 不做:
  - 目录迁移

### P-010 提交消息与命名规律

- 做:
  - 命名规则样本
  - 提交消息模板约束
  - 不自动升版护栏
- 不做:
  - 提交消息自动生成器

## 五、实施阶段

### Phase 1: 锁定范围与版本真源

#### 目标

- 锁定 `v1.0.0` 的实施边界
- 锁定版本字段和版本更新规则

#### 输出物

- 本文档 `[v1.0.0-plan.md](./v1.0.0-plan.md)`
- `v1.0.0` 的固定范围定义

#### 主要工作

- 明确 `P0` 全量项和 `P1` 最小护栏项
- 固定 `project_version = 1.0.0` 的机器真源格式
- 固定“默认不自动升版, 仅人工确认后修改真源”的规则
- 明确本版不做项:
  - 多 agent 运行时能力
  - Git Hook 自动安装
  - 根目录迁移
  - 自动版本 bump
  - 提交消息自动生成器

#### 完成标准

- `P0` / `P1` 的实施边界已经写清
- `project_version` 格式已经锁定为 `1.0.0`
- 后续 Phase 无需再重新讨论本版范围

### Phase 2: 落地嵌入式骨架与前置规则

#### 目标

- 提供可初始化的嵌入式治理骨架
- 把高频前置条件提前写入模板

#### 输出物

- `init --project-type embedded` 生成骨架
- 嵌入式默认模板
- 骨架基线说明

#### 主要工作

- 为 `embedded` 项目生成默认文档骨架:
  - `README.md`
  - `docs/DEVELOPMENT_PLAN.md`
  - `docs/PROTOCOL_SPEC.md`
  - `docs/HARDWARE_BRINGUP.md`
  - `docs/VALIDATION_PLAN.md`
- 在 profile 或模板中增加嵌入式最小前置字段:
  - 通信模式
  - 同步 / 异步模型
  - 波特率
  - Wi-Fi 频段限制
  - IP 策略
  - 引脚映射占位
  - 板载灯 / 烧录接口占位
  - 阈值 / 校准占位
- 提供默认目录规则, 但允许覆盖
  - 默认值可以是 `src + lib`
  - 允许覆盖为 `src + include`、`core + drivers` 等项目习惯

#### 完成标准

- `init --project-type embedded` 能生成完整骨架
- 骨架包含所有必需文档与前置占位
- 默认结构和可覆盖规则均已写清
- **Phase 2 完成后冻结骨架结构, 作为 Phase 3 校验的输入基线**

### Phase 3: 扩展 validate 与 P0 校验闭环

#### 目标

- 让 `validate` 能对嵌入式项目做最小可执行校验
- 把验证链路收成固定模板

#### 输出物

- 扩展后的 `validate`
- `docs/VALIDATION_PLAN.md` 模板规范

#### 主要工作

- 把 `M0 / M1 / M2 / M3` 作为文档模板生成到 `docs/VALIDATION_PLAN.md`
- `validate` 至少检查 `docs/VALIDATION_PLAN.md` 是否已生成且包含 `M0 / M1 / M2 / M3` 四段结构
- 明确其职责:
  - `M0`: 文档、版本、参数占位齐全
  - `M1`: Mock 路径和计划存在
  - `M2`: 协议文档、Schema 示例、测试计划存在且版本一致
  - `M3`: 真机握手、主链路、验收项已定义
- `validate` 本版至少检查:
  - 关键嵌入式字段是否存在
  - 必需文档是否存在
  - README 是否链接关键文档
  - `project_version` 格式是否正确
  - 协议文档、Schema 示例、测试计划版本号是否一致
  - 中文 UTF-8 是否存在明显异常
- 把“Contract-First 最小实现闭环”收敛为:
  - 协议文档存在
  - Schema 示例存在
  - 测试计划存在
  - 上述内容与 `project_version` 一致

#### 完成标准

- `validate` 能覆盖本版所有 `P0` 校验项
- `docs/VALIDATION_PLAN.md` 已生成并纳入检查
- 缺字段、缺文档、版本格式错误、版本不一致都能报错

### Phase 4: 落地 P1 最小护栏

#### 目标

- 补齐最小同步能力和最小治理护栏
- 防止后续继续失控

#### 输出物

- 版本规则说明
- 根目录导航与目录说明
- 命名规则和提交消息约束

#### 主要工作

- `P-004`
  - 让所有受管输出统一引用 `project_version`
  - 明确默认不自动升版
- `P-006`
  - 固化 Windows / PowerShell / UTF-8 / pytest 临时文件规则说明
  - 纳入最小校验护栏
- `P-009`
  - 更新根目录阅读顺序
  - 更新目录说明
  - 明确“入口 / 真源 / 适配输出”的分类
- `P-010`
  - 整理命名样式样本
  - 固定“版本 + 阶段 + 动作 + 目标”的推荐表达
  - 提交消息继续采用 Conventional Commit, 但要贴合项目中文语义

#### 完成标准

- 版本号只引用 `project_version`
- 根目录入口与目录说明一致
- 命名规则与提交消息模板已经可执行
- `P1` 最小护栏没有超范围膨胀

### Phase 5: 回归、验收与发布收口

#### 目标

- 验证本版能力是否真正闭环
- 为发布和沉淀做收口

#### 输出物

- 测试通过记录
- 发布收口记录
- 本版 progress entry

#### 主要工作

- 运行并通过:
  - `python -m vibe_governance render --target .`
  - `python -m vibe_governance validate --target .`
  - `python -m unittest discover -s tests -v`
- 文档一致性检查:
  - 升级文档路径正确
  - 根目录说明与实际结构一致
  - 中文内容无乱码
- 成功标准逐条验收:
  - 成功标准 1: `embedded` 骨架可生成
  - 成功标准 2: 嵌入式项目包含通信模式、关键硬件前提和验证路径
  - 成功标准 3: `validate` 可发现缺失与不一致
  - 成功标准 4: 目录说明、命名规则、提交消息约束已存在
- 每个 Phase 完成后保留可回退锚点, 优先用 Git tag, 若不打 tag 则至少保留明确提交点

#### 成功标准映射

- 成功标准 1 对应:
  - `init --project-type embedded` 生成成功
  - 骨架包含必需文档
- 成功标准 2 对应:
  - profile / 模板中存在通信模式、关键硬件前提和验证路径占位
  - `validate` 能检查这些字段是否存在
- 成功标准 3 对应:
  - `validate` 能发现缺字段、缺文档、版本格式错误、版本不一致
  - 版本号默认不自动升版
- 成功标准 4 对应:
  - 根目录说明、命名规则、提交消息约束已落盘
  - 升级文档路径与阅读顺序可被新接手者快速理解

#### 完成标准

- 代码、模板、文档、测试全部通过
- 4 条成功标准均有对应验收结果
- 已保留可回退锚点和发布收口记录

## 六、测试计划

### 正常场景

- `init --project-type embedded` 成功生成完整骨架
- `render` 对同一输入保持确定性
- `validate` 对完整嵌入式骨架返回通过

### 失败场景

- `init --project-type invalid` 报错
- `project_version` 缺失时报错
- `project_version` 不是 `major.minor.patch` 时报错
- 缺少关键文档时报错
- 协议文档、Schema 示例、测试计划版本不一致时报错
- README 未链接关键文档时报错
- 中文编码出现明显异常时报错
- `validate` 同时发现多个错误时输出聚合错误信息

### 人工检查项

- 文件路径正确: `docs/upgrades/v1.0.0-plan.md`
- 与 `v1.0.0-summary.md` 分工清晰, 不混写问题真源内容
- 正文已包含:
  - `P1` 范围清单
  - `project_version` 格式定义
  - 每个 Phase 的完成标准
  - `M0 / M1 / M2 / M3` 的落点
  - 回滚锚点说明

## 七、假设与本版不做项

### 假设

- 本轮“只落盘实施计划”理解为: 仅新增 `v1.0.0-plan.md`
- `v1.0.0-summary.md` 暂时不承载实施计划正文
- 当前版本号仍保持 `v1.0.0`, 本次落盘不触发升版
- 每个 Phase 完成后优先打 Git tag 作为回滚锚点; 若当轮不打 tag, 则至少保留明确提交点用于回退

### 本版不做项

- 多 agent 运行时平台能力
- 自动版本 bump
- Git Hook 自动安装
- 根目录目录迁移
- 提交消息自动生成器
