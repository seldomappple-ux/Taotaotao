# v1.2.0 实施计划

这份文档是 `v1.2.0` 的执行计划正文。

它只负责说明怎么实施, 不承担问题归纳职责。

相关文档分工如下:

- `v1.2.0-summary.md`: 问题总结与版本边界
- `v1.2.0-plan.md`: 实施步骤与验收方式

## 一、版本目标

`v1.2.0` 只解决 4 件事:

1. 固定唯一增量真源 `docs/DELTA_DECISIONS.md`
2. 固定纯索引文件 `docs/NEXT_ITERATION_BASELINE.md`
3. 固定“是否影响跨层契约”的判定与回写规则
4. 保持根目录和目录结构的人读友好

## 二、关键接口与文档职责

### 1. `docs/DELTA_DECISIONS.md`

职责:

- 作为唯一增量真源
- 记录当前有效决议
- 记录已升格和已废弃状态

字段固定为:

- `delta_id`
- `status`
- `level`
- `created`
- `expires`
- `promotion_target`
- `source_entries`

### 2. `docs/NEXT_ITERATION_BASELINE.md`

职责:

- 作为下一轮生成前的纯索引文件

允许包含:

- 当前版本
- 必读文件路径
- 文件作用说明
- 阅读顺序
- 当前有效 `delta_id`
- 固定回归检查项

不得包含:

- 决议正文摘要
- 本轮目标正文
- 设计内容复述

### 3. 判定规则

唯一判断轴:

- 是否影响跨层契约

#### 判定决策树

1. 是否在协议文档 / schema / 硬件清单 / 目录约定中明文规定?
   - 是 -> `L2`
   - 否 -> 继续判断

2. 是否改变外部可观察行为?
   - 包括:
     - 错误码
     - 状态转换
     - 超时表现
     - 前端消费语义
     - 硬件前提
   - 是 -> `L2`
   - 否 -> 继续判断

3. 是否影响下一轮 AI 对系统行为的理解?
   - 是 -> `L1`
   - 否 -> `L0`

关键原则:

- 有疑问时, 优先判为更高级别
- 宁可多回写, 不要漏回写

#### 目标

#### 输出物

#### 完成标准

### Phase 3.5: 快速验证

#### 目标

- 用真实案例小规模验证规则是否好用

#### 输出物

- 2 个历史案例的模拟分流结果
- 必要的规则或模板微调

#### 主要工作

- 从 `.agents/progress/entries/` 选 2 个真实修复案例
- 至少覆盖:
  - 1 个 `L0 / L1`
  - 1 个 `L2`
- 模拟:
  - 按决策树判级
  - 填写 `DELTA_DECISIONS.md`
  - 更新 `NEXT_ITERATION_BASELINE.md`

#### 验证点

- 判定是否有歧义
- 7 个字段是否够用
- 纯索引是否清晰

#### 完成标准

- 小样本验证通过后再继续推进其余文档同步

### Phase 4: 补迁移路径与导航

#### 目标

- 让旧项目、新项目、首次接入项目都知道怎么使用新机制

- 更新后的 `README.md`

#### 主要工作

#### 完成标准

- 新老项目都能找到接入路径
- 根目录入口依然保持简洁

### Phase 5: 可选的索引刷新辅助

#### 目标

- 定义未来可选的最小自动化能力

#### 输出物

- `baseline refresh` 能力说明
- 自动区块边界约定

#### 主要工作

- 明确未来命令:
  - `vibe-governance baseline refresh`
- 说明它只做:
  - 扫描 `status=active` 的条目
  - 刷新 `NEXT_ITERATION_BASELINE.md` 中的自动区块
- 说明它不做:
  - 摘要生成
  - 正文修改
  - 人工判断替代

#### 完成标准

- 本能力可推迟到 `v1.3.0`
- 不阻塞 `v1.2.0` 发布

## 四、首个 DELTA 决议的触发时机

`v1.2.0` 发布时, `docs/DELTA_DECISIONS.md` 可以是空模板。

第一条决议的写入时机固定为:

1. `v1.2.0` 发布后的第一次修复
2. 按决策树判定为 `L1` 或 `L2`
3. 创建第一条 `delta_id`

示例:

- 如果发布后首次修复的是一个网关 ACK 超时问题
- 且该变化影响跨层契约
- 则创建第一条 `L2` 决议
- 回写真源后将其标记为 `promoted`

## 六、测试计划

### 正常流程验证

- `DELTA_DECISIONS.md` 是唯一增量真源
- `NEXT_ITERATION_BASELINE.md` 是纯索引
- `L1` 条目必须有 `created` 和 `expires`
- 连续 2 个迭代仍依赖的 `L1` 会触发升格或删除
- 根目录仍保持 5 个入口文件

### 异常流程验证

1. 误判测试
   - 故意把一个协议字段变化判为 `L1`
   - 预期: 人工 review 能识别它其实属于 `L2`

2. 遗漏回写测试
   - 创建一个 `L2` 但不回写真源
   - 预期: 合并前检查清单会拦住

3. 垃圾场测试
   - 模拟累积 20 个未清理的 `L1`
   - 预期: `next_cleanup_due` 会提示需要清理

4. 回退测试
   - 模拟一次 `L2` 重跑后方向错误
   - 预期: 存在回退锚点, 可以回退
