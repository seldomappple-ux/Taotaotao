# v1.0.0 升级计划评价

评价日期: 2026-03-30
评价对象: docs/upgrades/v1.0.0-summary.md 中的"v1.0.0 项目升级计划"章节

## 一、整体评价

**定位清晰度:** ✅ 优秀
- "P0 全量落地 + P1 最小护栏"定位明确
- 4 条成功标准具体可验证

**范围控制:** ✅ 合理
- 明确不追求一次做满
- 多 agent 研究不纳入本版交付

## 二、发现的缺点和漏洞

### 缺点 1: Phase 划分与实际依赖关系不匹配 (中等严重)

**问题描述:**
- Phase 2 要落地嵌入式骨架
- Phase 3 要扩展 validate 检查嵌入式项目
- **但 Phase 3 的检查依赖 Phase 2 的骨架结构**

**影响:**
如果 Phase 2 的骨架结构变化,Phase 3 的检查逻辑也要跟着改,但计划中没有明确这个依赖。

**建议修正:**
在 Phase 2 末尾补充:"完成后冻结骨架结构,作为 Phase 3 校验的输入基线"

### 缺点 2: "P1 最小护栏"范围不明确 (高严重)

**问题描述:**
- Summary 说"P1 最小护栏"
- Phase 4 标题是"做完 P1 最小护栏"
- **但没有明确说 P-004/P-006/P-009/P-010 各做到什么程度**

**影响:**
- P-004 版本同步: 做"版本依赖图"还是只做"版本号统一引用"?
- P-006 环境治理: 做"Git Hook 模板"还是只做"规则文档"?
- P-009 根目录治理: 做"目录迁移"还是只做"文档说明"?
- P-010 (文档中未定义此问题)

**建议修正:**
在 Phase 1 补充明确的 P1 范围清单,例如:
```
P-004: 只做版本号统一引用,不做依赖图
P-006: 只做规则文档,不做 Git Hook
P-009: 只做文档说明,不做目录迁移
```

### 缺点 3: P-010 未定义但被引用 (高严重)

**问题描述:**
- Phase 1 提到"P-004/P-006/P-009/P-010"
- **但 v1.0.0-summary.md 只定义了 P-001 到 P-009**
- P-010 不存在

**影响:**
执行时会困惑 P-010 是什么,导致范围不清。

**建议修正:**
删除 P-010 引用,或在 v1.0.0-summary.md 中补充 P-010 定义。

### 缺点 4: Phase 2 嵌入式骨架内容过于具体,缺少灵活性 (中等严重)

**问题描述:**
- 固化了 `.c` 进 `src/`, `.h` 进 `lib/`
- 固化了"新模块成对创建 `.c + .h`"
- **但不同嵌入式项目的目录约定可能不同**

**影响:**
- Arduino 项目习惯 `src/` 和 `include/`
- PlatformIO 项目习惯 `src/` 和 `lib/`
- 裸机项目可能习惯 `core/` 和 `drivers/`

**建议修正:**
改为"提供默认骨架,允许通过 profile 覆盖目录约定"

### 缺点 5: Phase 3 的 M0/M1/M2/M3 验收模板缺少实施细节 (中等严重)

**问题描述:**
- 提到"把 Moss_Q 的分阶段验收思路模板化为 M0/M1/M2/M3"
- **但没有说明如何模板化,存放在哪里,如何触发**

**影响:**
- 不清楚是生成 Markdown 模板还是代码检查
- 不清楚是手动执行还是自动校验
- 不清楚与 `validate` 命令的关系

**建议修正:**
补充说明:"M0-M3 作为文档模板生成到 `docs/VALIDATION_PLAN.md`,由 validate 命令检查是否存在"

### 缺点 6: "Contract-First 最小实现闭环"定义模糊 (中等严重)

**问题描述:**
- Phase 3 提到"协议真源、Schema/示例、实现、测试四者必须可被同一套规则校验"
- **但没有说明"同一套规则"是什么,如何校验**

**影响:**
- 不清楚是文件存在性检查还是内容一致性检查
- 不清楚校验规则写在哪里
- 不清楚如何判断"一致"

**建议修正:**
改为"协议文档、Schema 示例、测试计划必须存在且可被 validate 检查到,版本号必须一致"

### 缺点 7: Test Plan 缺少失败场景测试 (低严重)

**问题描述:**
- Test Plan 只列了正常流程的测试
- **缺少异常场景测试**

**影响:**
- 不知道错误提示是否友好
- 不知道边界条件是否处理正确

**建议补充:**
- `init` 缺少 `--project-type` 时的提示
- `validate` 发现多个错误时的输出格式
- `render` 遇到冲突时的处理方式
- 版本号格式错误时的报错

### 漏洞 1: Phase 1 缺少"锁定 P0/P1 具体范围"的输出物 (高严重)

**问题描述:**
- Phase 1 说"明确 v1.0.0 的交付边界"
- **但没有说明这个"明确"的结果写在哪里**

**影响:**
- Phase 2-4 执行时不知道参考哪份文档
- 容易在执行过程中范围蔓延

**建议修正:**
在 Phase 1 补充:"输出 `docs/upgrades/v1.0.0-scope.md`,明确列出每个问题的实施范围"

### 漏洞 2: 缺少 Phase 之间的验收标准 (高严重)

**问题描述:**
- 5 个 Phase 之间没有明确的验收标准
- **不知道什么时候可以进入下一个 Phase**

**影响:**
- Phase 2 做到什么程度才能进 Phase 3?
- Phase 3 做到什么程度才能进 Phase 4?
- 容易出现"Phase 2 没做完就开始 Phase 3"的情况

**建议修正:**
每个 Phase 末尾补充"Phase N 完成标准",例如:
- Phase 2 完成标准: 嵌入式骨架可通过 `init --project-type embedded` 生成,且包含所有必需文档
- Phase 3 完成标准: `validate` 可检查嵌入式项目的所有 P0 问题

### 漏洞 3: 缺少回滚策略 (中等严重)

**问题描述:**
- 计划中没有提到如果某个 Phase 失败怎么办
- **没有回滚或降级方案**

**影响:**
- Phase 3 做到一半发现 Phase 2 骨架有问题,如何回退?
- Phase 5 验证失败,是回退到哪个 Phase?

**建议修正:**
在 Assumptions 中补充:"每个 Phase 完成后打 Git tag,失败时可回退到上一个 Phase 的 tag"

### 漏洞 4: Public Interfaces 中 `project_version` 字段格式未定义 (中等严重)

**问题描述:**
- 说 `project_version` 是必填字段
- **但没有说明格式要求**

**影响:**
- 不知道是 `1.0.0` 还是 `v1.0.0`
- 不知道是否支持 `-alpha` 等后缀
- 校验时无法判断格式是否正确

**建议修正:**
补充:"格式为 `major.minor.patch`,例如 `1.0.0`,不带 `v` 前缀"

### 漏洞 5: Phase 5 验证项与 4 条成功标准不完全对应 (低严重)

**问题描述:**
- Summary 定义了 4 条成功标准
- Phase 5 定义了验证项
- **两者不完全对应**

**影响:**
- 成功标准 1: "能初始化带嵌入式治理骨架的新项目" → Phase 5 有对应 (CLI 回归)
- 成功标准 2: "能在启动前锁定软硬件链路" → Phase 5 缺少明确验证
- 成功标准 3: "能做机器校验" → Phase 5 有对应 (Validate 校验)
- 成功标准 4: "能收敛到可持续维护状态" → Phase 5 缺少明确验证

**建议修正:**
在 Phase 5 补充:
- 软硬件链路验证: 嵌入式项目必须包含通信模式、端口分配等关键字段
- 可持续维护验证: 文档路径、命名规则、提交消息示例符合规范

## 三、严重程度汇总

### 高严重 (3个)
1. P-010 未定义但被引用
2. "P1 最小护栏"范围不明确
3. 缺少 Phase 之间的验收标准

### 中等严重 (5个)
1. Phase 划分与依赖关系不匹配
2. 嵌入式骨架过于具体
3. M0/M1/M2/M3 缺少实施细节
4. Contract-First 定义模糊
5. `project_version` 格式未定义

### 低严重 (2个)
1. Test Plan 缺少失败场景
2. Phase 5 验证项与成功标准不完全对应

## 四、建议修正优先级

### 必须修正 (阻塞实施)
1. 删除或定义 P-010
2. 明确 P1 范围清单
3. 补充 Phase 验收标准
4. 定义 `project_version` 格式

### 建议修正 (提高质量)
1. 补充 Phase 1 输出物
2. 调整骨架灵活性
3. 明确 M0-M3 实施方式
4. 简化 Contract-First 定义
5. 补充回滚策略

### 可选修正 (锦上添花)
1. 补充失败场景测试
2. 对齐验证项与成功标准

## 五、总体评价

**计划完整性:** ⚠️ 基本完整,但有关键细节缺失
**可执行性:** ⚠️ 需要修正高严重问题后才可执行
**风险控制:** ⚠️ 缺少回滚策略和 Phase 验收标准

**结论:** 计划方向正确,但需要补充细节后才能开始实施。
