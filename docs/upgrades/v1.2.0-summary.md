# v1.2.0 升级总结

这份文档是 `v1.2.0` 的问题真源与版本总结。

它负责说明:

- 本轮为什么要做
- 本轮要解决什么问题
- 本轮采用什么治理口径
- 本轮实施边界是什么

它不负责展开实施步骤。实施步骤请看 [v1.2.0-plan.md](./v1.2.0-plan.md)。

## 一、版本定位

`v1.2.0` 不是功能型版本, 而是一次迭代治理机制升级。

本轮要解决的核心矛盾是:

- 上游架构 / 协议必须稳定, 不能被每次局部修复污染
- 修复后的真实变化不能只留在代码和聊天里
- 下一轮 AI 生成不能继续只读旧真源
- 人查看文件时, 目录和入口必须保持简洁

## 二、本轮要解决的 4 个问题

### P-001 缺少唯一增量真源

- 现象:
  - 修复后常常只有代码变了, 但没有中间层承接
  - “暂不回写上游”的变化没有稳定落点
- 风险:
  - 下一轮 AI 会继续基于旧真源生成
  - 修复事实在多轮迭代后逐渐丢失
- 本轮决策:
  - 新增 `docs/DELTA_DECISIONS.md`
  - 固定它是唯一增量真源

### P-002 下一轮生成输入缺少稳定入口

- 现象:
  - 下一轮迭代时, 人和 AI 都要重新拼接“这次该读哪些文件”
- 风险:
  - 输入包不稳定
  - 同一轮不同人 / 不同 AI 读取口径不一致
- 本轮决策:
  - 新增 `docs/NEXT_ITERATION_BASELINE.md`
  - 固定它是纯索引文件, 不重复保存内容正文

### P-003 修复后是否要回写上游没有统一规则

- 现象:
  - 同一类变化, 不同人可能做出不同判断
  - 过度回写会污染上游, 不回写又会造成漂移
- 风险:
  - 架构 / 协议与实现长期偏离
  - 修复分类成本过高
- 本轮决策:
  - 固定唯一判断轴: 是否影响跨层契约
  - 固定 `L0 / L1 / L2` 分级
  - 固定判定决策树和人工门禁

### P-004 人读友好正在被文档膨胀侵蚀

- 现象:
  - 一旦治理文档继续增长, 根目录和 `docs/` 容易再次混乱
- 风险:
  - 人很难快速定位当前应该看的文件
  - 维护者误以为要记两套体系
- 本轮决策:
  - 坚持单体系文件结构
  - 根目录继续只保留 5 个入口文件
  - 所有新增治理内容全部下沉到 `docs/`

## 三、本轮固定口径

### 1. 单体系, 不做两套阅读体系

- 人和 AI 使用同一套文件结构
- 不同点只在读取深度, 不在文件体系

### 2. `DELTA_DECISIONS.md` 是唯一增量真源

- 所有仍在生效的增量决议只能落这里
- 其他文件不得重复保存决议正文

### 3. `NEXT_ITERATION_BASELINE.md` 是纯索引

- 只列:
  - 文件路径
  - 一句话用途
  - 建议阅读顺序
  - 当前有效 `delta_id`
  - 固定回归检查项
- 不写摘要
- 不写正文解释

### 4. 唯一判断轴: 是否影响跨层契约

如果影响:

- 判为 `L2`
- 必须回写对应真源

如果不影响, 再判断:

- 是否会影响下一轮 AI 对系统行为的理解
  - 是: `L1`
  - 否: `L0`

### 5. `L1` 必须有限期

- 每个 `L1` 最多存活 2 个迭代周期
- 到期后必须升格、废弃或移除

## 四、本轮不做项

本轮明确不做:

- 第二套“人读专用”文档体系
- `REQUIREMENTS_BASELINE.md`
- 强制机器判定 `L0 / L1 / L2`
- `baseline refresh` 命令实现
- 复杂自动审批流

## 六、Phase 3.5 快速验证结果

本轮已用 2 个真实历史案例试跑判定决策树:

### 案例 A: `2026-03-31-1`

- 来源: `.agents/progress/entries/2026/2026-03-31-1.md`
- 变更内容:
  - 引入 `project_version`
  - 新增 `embedded init`
  - 新增嵌入式必需文档
  - 扩展 `validate`
- 判定结果: `L2`
- 原因:
  - 直接影响 profile schema、文档结构和外部可观察校验行为
  - 已属于跨层契约变化

### 案例 B: `2026-03-31-3`

- 来源: `.agents/progress/entries/2026/2026-03-31-3.md`
- 变更内容:
  - 新增根目录物理治理清单
  - 将测试临时目录下沉到 `tests/.tmp-tests/`
- 判定结果: `L1`
- 原因:
  - 会影响后续人和 AI 对目录整洁与入口结构的理解
  - 但没有改动正式协议、schema 或硬件契约

当前结论:

- 判定决策树足以覆盖治理仓与嵌入式项目的两类典型场景
- `L0 / L1 / L2` 三档目前够用
