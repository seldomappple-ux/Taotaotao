这份文档用于承接 [UPGRADE_FEEDBACK.md](./UPGRADE_FEEDBACK.md) 里的原始记录, 进行后续的总结、归纳和问题拆分。

- `UPGRADE_FEEDBACK.md`: 只保留原始记录
- `UPGRADE_SUMMARY.md`: 负责总结、归类、优先级和行动项

## 当前状态

当前已开始面向 `v1.0.0` 做第一轮问题提炼。

本轮目标不是把所有细节一次性定完, 而是先从大量原始记录里识别出真正的治理缺口, 避免后续继续按零散 bug 的方式修修补补。

**升级启动日期**: 2026-03-30

## 本轮升级范围

- 目标版本: `v1.0.0`
- 输入来源:
  - [UPGRADE_FEEDBACK.md](./UPGRADE_FEEDBACK.md) 中的用户口述嵌入式反馈
  - 导入的 `Moss_Q` progress 原始记录
- 当前阶段: 第一轮问题归纳

## 先说结论

从原始记录看, 这轮升级真正要解决的, 不是某几个单独的 bug, 而是下面 5 类根问题:

1. 嵌入式项目的初始化骨架还不够完整, 很多基础约束没有前置.
2. 软硬件一体化架构的前置设计不足, 导致通信、状态机和联调问题反复出现.
3. 文档、协议、实现、版本之间缺少稳定同步机制, 容易越做越分叉.
4. 验证与环境治理能力不足, 导致 mock、真机、编码、脚本和测试经常互相打架.
5. 根目录与说明文档入口仍然偏杂乱, 影响接手和升级导航效率.

也就是说, 原始记录里大量看似分散的问题, 本质上大多不是“单点实现太差”, 而是“前置治理不够”和“闭环不完整”。

## 如何理解原始记录和真正问题的关系

原始记录大致可以分成 3 层:

### 第 1 层: 现象层

例如:

- 波特率没配好导致串口不能用
- TCP / UDP 行为不一致
- 校准阈值过紧
- README 没更新
- PowerShell 中文乱码
- mock 路径和真机路径不一致
- 根目录一眼看不出哪些是入口、哪些是真源、哪些是适配输出

这些是你在实际开发中最先感受到的“痛点”。

### 第 2 层: 修补层

例如 `Moss_Q` 里大量 progress entry 记录的是:

- 某个 ACK 路由修了
- 某个状态字段对齐了
- 某个校准阈值调整了
- 某个 mock 死锁解开了
- 某个 README 或协议文档补齐了

这些很重要, 但它们多数是在给现象打补丁。

### 第 3 层: 治理层

真正决定 `v1.0.0` 是否升级成功的, 是能不能把前两层收敛成稳定能力:

- 初始化时就把基础约束写全
- 联调前就把架构边界定清
- 协议、文档、实现不再长期漂移
- mock / 真机 / 脚本 / 编码 / 测试形成一致闭环

所以本轮 summary 的重点, 不是统计 bug 数量, 而是把“重复返工的根因”找出来。

## v1.0.0 核心问题清单

### P-001 嵌入式项目初始化骨架不完整

- 来源批次:
  - 用户口述 `2026-03-12` 至 `2026-03-21`
  - `Moss_Q` 中关于目录、硬件接线、板载灯、版本与 README 的多条记录
- 优先级: `P0`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 生成器代码层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - 新开嵌入式项目时, 根目录容易变复杂.
  - `.c / .h` 文件组织、模块生成规则、开发计划、网关文档、协议文档位次没有在一开始定清.
  - 板载灯、烧录接口、硬件接线、README、版本号等常用内容缺少默认骨架.
- 根因判断:
  - 当前治理内核只搭好了通用治理地基, 还没有正式的嵌入式 overlay 和初始化模板层.
- 风险:
  - 每个项目都要从头补目录约定和基础文档, 极易把经验散落成口头知识.
- 相关约束:
  - 当前仓库明确说明 overlay 和嵌入式规则包还只是预留, 并未正式实现.

#### 建议动作

- 文档层:
  - 定义嵌入式项目目录、README、计划文档、协议文档的标准骨架.
- `.agents` 真源层:
  - 补嵌入式项目的事实字段和项目约束样例.
- 生成器代码层:
  - 扩展 profile schema / scaffold / validator.
- 模板 / 规则层:
  - 增加嵌入式初始化模板和强制性前置检查项.

#### 决策状态

- `planned`

### P-002 软硬件一体化架构前置设计不足

- 来源批次:
  - 用户口述关于 TCP/UDP、长短连接、同步异步、NVLink/MVLink、上下位机一体化设计的反馈
  - `Moss_Q` 中关于 ACK 路由、UDP 端口、状态机、协议收口的大量记录
- 优先级: `P0`
- 影响层级:
  - `.agents` 真源层
  - 文档层
  - 生成器代码层

#### 问题说明

- 现象:
  - 通信方式在项目中途才讨论, 导致 TCP/UDP、ACK 路由、长短连接、重试和状态恢复反复返工.
  - 上位机、下位机、前端和协议真源经常不是按同一条数据链条一起设计.
- 根因判断:
  - 当前流程里缺少”软硬件架构一体化前置确认”, 导致协议设计、状态机、端口分工、同步/异步模型没有在启动前锁定.
  - **深层根因**: 缺少协议设计规范 (端口分配、回包路径、重复投递处理) 和状态一致性保证机制 (capture_token 同步、status.connected 同步、三端状态同步).
- 风险:
  - 真机联调时容易把”协议问题、时序问题、网络问题、状态问题”混成一团, 修复成本很高.
  - 状态不同步会导致前端断连保护与恢复选择逻辑失真.
- 相关约束:
  - 这类问题不适合靠后期文档补丁解决, 需要进入项目初始化和架构评审入口.
  - MVLink 单 UDP 时接收数据和控制指令必须拆到不同端口.
  - 同步/异步架构必须一开始决定, NVLink 类软硬件程序推荐异步架构.

#### 建议动作

- 文档层:
  - 增加”软硬件链路一体化设计清单”.
  - 补充协议设计规范: 端口分配策略、UDP 回包路径设计、重复投递处理.
- `.agents` 真源层:
  - 补通信模式、状态机策略、同步/异步模型等结构化字段或 ADR.
  - 增加状态同步一致性约束 (如三端状态同步延迟 < 200ms).
- 生成器代码层:
  - 为嵌入式项目提供前置检查入口.
- 模板 / 规则层:
  - 固化通信模式、端口设计、ACK 路由、状态恢复的红线规则.
  - 提供状态同步模式模板 (token 同步、连接状态同步).

#### 决策状态

- `planned`

### P-003 基础硬件配置没有被前置固化

- 来源批次:
  - 用户口述关于波特率、2.4GHz、固定 IP、噪声阈值、颜色校准、板载灯和接口预留的反馈
  - `Moss_Q` 中关于引脚映射、阈值调参、校准、灯语和硬件说明的记录
- 优先级: `P0`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - 一些最基础的硬件前提没有在项目开始时写明, 比如波特率、WiFi 频段、固定 IP、引脚映射、校准阈值、灯珠数量与供电约束.
- 根因判断:
  - 当前治理体系对“硬件启动前必须明确的基础条件”缺少 checklist 和模板化落点.
- 风险:
  - 很多时间会浪费在环境误配和基础前提缺失上, 甚至会把设备/协议问题误判成代码问题.
- 相关约束:
  - 这些项适合做成初始化向导、规则清单或强制模板, 不应继续靠临场补充.

#### 建议动作

- 文档层:
  - 制定“硬件 bring-up 最小清单”.
- `.agents` 真源层:
  - 增加硬件参数占位和项目级约束.
- 模板 / 规则层:
  - 嵌入式模板默认带出波特率、网络、引脚、阈值、灯语与校准占位.

#### 决策状态

- `planned`

### P-004 版本、README、文档和实现之间缺少统一同步机制

- 来源批次:
  - 用户口述关于版本号统一、README 自动更新、文档位次的反馈
  - `Moss_Q` 中关于多份计划文档、协议文档、README、版本升级和文档收口的记录
- 优先级: `P1`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - 版本号经常要靠人工记忆同步.
  - README 和落盘说明容易滞后.
  - 同一轮迭代里, 协议、实现、前端、网关和文档之间容易出现口径漂移.
- 根因判断:
  - 当前只有治理内核自己的版本闭环, 但还没有把”业务项目版本”和”文档/README 同步策略”抽象成模板能力.
  - **深层根因**: Moss_Q 已建立版本管理规则, 但多文件版本号的联动更新没有自动化检查. 协议版本、Schema、示例、文档之间的版本绑定关系缺少机器校验.
- 风险:
  - 后续接手的人会看到多份”都像是真的”文档, 但不知道哪份才是现行标准.
  - 每次主协议升版后需要手动检查 README、示例 JSON、版本变更记录三处是否一起更新.
- 相关约束:
  - 文档不能写得比实现超前, 已在外部项目记录中多次被证明是高风险点.

#### 建议动作

- 文档层:
  - 为 README、协议、计划文档建立统一更新规则.
  - 建立版本依赖图和影响范围分析文档.
- `.agents` 真源层:
  - 补版本和文档同步约束.
  - 记录协议版本与 Schema/示例/文档的绑定关系.
- 模板 / 规则层:
  - 设计 README/版本/归档说明的默认模板与更新触发条件.
  - 提供版本号联动检查工具 (检查协议版本、Schema、示例、文档是否一致).

#### 决策状态

- `planned`

### P-005 验证链路不完整, mock 与真机路径经常分叉

- 来源批次:
  - 用户口述关于 mock、前端测试、硬件握手测试的反馈
  - `Moss_Q` 中关于 mock 死锁、mock 参数不一致、真机联调和 pytest 缺口的记录
- 优先级: `P0`
- 影响层级:
  - 生成器代码层
  - 文档层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - mock 模式和真机模式经常不是同一条执行链.
  - pytest 缺少硬件握手、协议时序、网络状态切换等关键案例.
  - 很多问题只有在真机阶段才暴露.
- 根因判断:
  - 当前验证体系更偏治理内核健康检查, 还没有形成嵌入式项目的”mock -> 协议 -> 真机”分层验证闭环.
  - **深层根因**: 缺少分阶段验收标准模板 (M0/M1/M2/M3 各有明确指标), 缺少自动化测试与实现的同步更新机制. Moss_Q 已有完整的 M0-M3 验收标准, 但没有模板化.
- 风险:
  - 代码在本地看似可用, 一到设备或网关联调就失真.
  - 前端测试口径不跟随实现同步更新会导致假阴性.
- 相关约束:
  - mock 应与真实路径共享尽可能多的解析、状态机和并发模型.
  - 每次修改路由/状态机逻辑后必须同步更新对应测试.

#### 建议动作

- 文档层:
  - 建立分层测试策略说明.
  - 提供分阶段验收标准模板 (参考 Moss_Q 的 M0-M3 验收).
- 生成器代码层:
  - 为嵌入式项目预留测试骨架生成能力.
  - 提供测试同步检查工具.
- 模板 / 规则层:
  - 固化 mock、协议验证、真机验收三层测试规则.
  - 提供验收指标模板 (如丢包率、延迟、连续性).

#### 决策状态

- `planned`

### P-006 编码、脚本和环境兼容性治理不足

- 来源批次:
  - 用户口述关于 PowerShell 中文、UTF-8、pytest 临时文件和环境变化沉淀的反馈
  - `Moss_Q` 中关于编码修复、脚本阻塞、运行模式误判、乱码清理的记录
- 优先级: `P1`
- 影响层级:
  - 文档层
  - 生成器代码层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - PowerShell 管道容易出现中文乱码.
  - 脚本容易因为环境差异、启动参数或交互方式出问题.
  - pytest 和临时文件清理在 Windows 上表现不稳定.
- 根因判断:
  - 当前仓库虽然有治理约束, 但对”Windows + PowerShell + 中文 + 真机脚本”这一现实组合还没有形成完整的环境治理规则.
  - **深层根因**: Moss_Q 已经在用 Git Hook 做编码守卫 (has-utf8-bom 检查), 解决方案已存在, 需要的是模板化和推广.
- 风险:
  - 很多非业务问题会持续消耗联调时间, 还会污染记录质量.
- 相关约束:
  - 这类问题需要固化成工具链规则和检查动作, 不能只靠经验记忆.

#### 建议动作

- 文档层:
  - 增加编码与环境问题处理规范.
  - 提供 Git Hook 模板和使用说明.
- 生成器代码层:
  - 研究是否补脚本模板、编码检查或环境 guard.
  - 提供 Git Hook 自动安装能力.
- 模板 / 规则层:
  - 固化 UTF-8、PowerShell 输出、运行模式确认和临时文件策略.
  - 提供编码检查 Git Hook 模板 (参考 Moss_Q 的 has-utf8-bom).

#### 决策状态

- `planned`

### P-007 经验记录很多, 但缺少从“补丁记录”回收成“前置能力”的机制

- 来源批次:
  - 整体 raw 记录观察
  - `Moss_Q` 大量 progress entry 与本轮用户治理反馈的对照
- 优先级: `P0`
- 影响层级:
  - `.agents` 真源层
  - 文档层
  - 生成器代码层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - 外部项目里已经有很多高质量 progress, 但它们多数还是停留在“这次修了什么”.
  - 用户口述反馈则更多是在指出“以后一开始就该怎样”.
- 根因判断:
  - 当前最缺的不是更多 progress, 而是从 progress 中稳定提炼出初始化规则、架构约束、模板默认值和测试骨架的升级机制.
- 风险:
  - 经验会越积越多, 但新项目仍然重复踩老坑.
- 相关约束:
  - 这是 `v1.0.0` 最核心的治理问题之一, 因为它决定这次升级到底是在“修历史”还是在“建立新起点”.

#### 建议动作

- 文档层:
  - 维持 raw / summary / progress 三层分工.
- `.agents` 真源层:
  - 对明确采纳的问题补规则、ADR 或项目事实.
- 生成器代码层:
  - 为经验回流提供更稳定的入口.
- 模板 / 规则层:
  - 把高频经验固化为嵌入式初始化能力.

#### 决策状态

- `planned`

### P-008 协议定义与实现之间缺少一致性保证机制

- 来源批次:
  - Moss_Q 中关于协议真源、Schema、实现代码、测试用例同步的大量记录
  - M3.5 协议真源与 Schema 示例的版本绑定
  - 从"文档先行、实现滞后"转为"运行时产物与协议真源一致"
- 优先级: `P0`
- 影响层级:
  - `.agents` 真源层
  - 文档层
  - 生成器代码层
  - 模板 / 规则层

#### 问题说明

- 现象:
  - 协议设计文档、Schema 定义、实现代码、测试用例之间容易出现口径分叉.
  - 文档先行但实现滞后, 导致文档与实际行为不一致.
  - 每次修改路由/状态机逻辑后测试没有同步更新.
- 根因判断:
  - 缺少 Contract-First 的协议治理能力.
  - 协议定义、Schema、实现代码、测试用例之间没有机器可校验的一致性约束.
- 风险:
  - 文档与机器校验口径分叉, 导致集成时发现大量不兼容.
  - 运行时产物与协议真源不一致, 调试困难.
- 相关约束:
  - 这个问题横跨 P-002 (协议设计)、P-004 (版本同步)、P-005 (测试同步), 但本质上是独立问题.

#### 建议动作

- 文档层:
  - 建立 Contract-First 协议设计流程.
  - 定义协议真源、Schema、实现、测试的一致性要求.
- `.agents` 真源层:
  - 记录协议定义与实现的绑定关系.
- 生成器代码层:
  - 提供协议一致性校验工具.
  - 支持从协议定义生成 Schema、测试骨架.
- 模板 / 规则层:
  - 固化协议变更时的同步检查清单.

## 深层问题总结

通过对照原始记录和 Moss_Q progress 条目, 发现当前问题总结的几个不足:

1. **P-002 的根因不是”讨论太晚”**, 而是缺少协议设计规范 (端口分配、回包路径、重复投递) 和状态一致性保证机制.
2. **P-004 的根因不是”人工记忆”**, Moss_Q 已有版本管理规则, 真正缺的是多文件版本号联动检查和版本依赖图.
3. **P-005 的根因不是”mock 分叉”**, 而是缺少分阶段验收标准模板 (Moss_Q 的 M0-M3 已经很完整) 和测试同步机制.
4. **P-006 的解决方案已存在**, Moss_Q 已在用 Git Hook 做编码守卫, 需要的是模板化和推广.
5. **缺少 P-008**, 协议定义与实现一致性是独立的根问题, 横跨多个其他问题但本质不同.

这些深层问题说明: **v1.0.0 的核心任务不是从零设计新能力, 而是把 Moss_Q 已经验证过的解决方案提取、模板化、工具化.**

## v1.0.0 核心要求 (2026-03-30 补充)

基于实际使用反馈, v1.0.0 必须满足以下三个核心要求:

### 要求 1: 全面版本号管理

**要求:**
- 所有文件 (包括生成的文件) 都必须加上版本号
- 用版本号管理所有可交付对象
- 版本号必须可追溯、可对比

**影响范围:**
- 适配器生成文件 (CLAUDE.md, GEMINI.md 等)
- 模板文件
- 规则文件
- 文档文件
- 配置文件

**实施要点:**
- 在文件头部或 frontmatter 中明确标注版本号
- 版本号格式统一 (建议 V主版本.次版本.修订号)
- 生成时自动带出版本号
- 提供版本对比工具

### 要求 2: 引入校验能力

**要求:**
- 新项目打开或旧项目引入治理体系后, 需要有明确的校验流程
- 校验内容: 生成完成、可正常使用、逻辑无问题
- 校验流程必须可被 AI 执行, 且尽量节省 TOKEN

**目标:**
- 不能每次都让 AI 自己摸索校验方法 (太耗 TOKEN)
- 提供标准化的校验命令和检查清单
- 校验结果清晰、可操作

**实施要点:**
- 扩展 `validate` 命令, 增加更多检查项
- 提供 `health-check` 或 `doctor` 命令
- 输出明确的通过/失败状态和修复建议
- 提供校验清单文档, AI 可直接引用

### 要求 3: 根目录干净整洁

**要求:**
- 根目录不要出现 `.cursor`、`.claude` 等 IDE 专属目录
- 这些目录统一放到一个管理目录中
- 根目录保持简洁, 便于人类和 AI 快速定位

**目标:**
- 根目录只保留必要的入口文件和核心目录
- IDE 适配文件集中管理
- 降低根目录复杂度

**实施要点:**
- 评估是否可以把 `.cursor`、`.claude` 等移到 `.agents/adapters/` 或类似位置
- 如果 IDE 强制要求根目录, 则在文档中明确说明哪些是必须的, 哪些可以隐藏
- 提供根目录清理指南
