# v1.0.0 升级总结

这份文档是 `v1.0.0` 的问题真源, 负责承接原始记录、提炼问题、明确边界和固定本版约束。

它的职责是:

- 记录 `v1.0.0` 的问题清单
- 说明每个问题的现象、根因、风险和影响层级
- 明确 `P0 / P1` 的归属
- 固定本版只做的 `P1` 最小护栏边界
- 固定本版接口约束和阶段完成判定

它不承担实施步骤编排, 实施细节请看   [v1.0.0-plan.md](./v1.0.0-plan.md)。

## 二、本版定位

`v1.0.0` 的定位不是“嵌入式能力全部做完”, 而是:

- 从通用治理原型升级到“可正式承接嵌入式治理项目”的第一版
- 先把高频返工点前置成模板、规则、检查和流程
- 不直接进入多 agent 运行时平台建设

本版收口线固定为:

- `P0`: 本版必须完成
- `P1`: 本版只做最小护栏

## 三、核心判断

从原始记录看, 这轮升级真正要解决的不是零散 bug, 而是 4 类结构性缺口:

1. 初始化骨架和硬件前置条件没有被模板化.
2. 软硬件链路和协议边界没有在项目启动前被锁定.
3. 版本、文档、协议、测试之间缺少统一同步口径.
4. 根目录、命名和提交消息缺少持续治理, 容易越用越乱.

## 四、问题清单

### P-001 嵌入式项目初始化骨架不完整

- 优先级: `P0`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 生成器代码层
  - 模板 / 规则层
- 现象:
  - 新开嵌入式项目时, 根目录和基础文档很快变复杂.
  - `.c / .h` 组织、模块生成规则、计划文档、协议文档没有默认骨架.
- 根因:
  - 当前仓库只有通用治理地基, 还没有正式的 `embedded` 初始化骨架.
- 风险:
  - 每个项目都要从头补约定, 经验只能停留在口头.
- 本版处理:
  - 本版纳入 `P0`, 需要提供默认骨架和最小前置占位.

### P-002 软硬件一体化架构前置设计不足

- 优先级: `P0`
- 影响层级:
  - `.agents` 真源层
  - 文档层
  - 生成器代码层
- 现象:
  - TCP/UDP、长短连接、同步/异步、ACK 路由、端口分工经常在中途才讨论.
- 根因:
  - 启动阶段缺少软硬件链路一体化设计清单.
- 风险:
  - 真机联调时协议问题、网络问题、状态问题混在一起.
- 本版处理:
  - 本版纳入 `P0`, 需要把通信模式、状态一致性和协议前置约束写进模板与校验口径.

### P-003 基础硬件配置没有被前置固化

- 优先级: `P0`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 模板 / 规则层
- 现象:
  - 波特率、Wi-Fi 频段、固定 IP、引脚映射、阈值、板载灯等最基础前提没有提前写明.
- 根因:
  - 缺少硬件 bring-up 最小清单.
- 风险:
  - 环境误配会被误判成代码问题.
- 本版处理:
  - 本版纳入 `P0`, 需要为硬件参数建立占位与最小清单.

### P-004 版本、README、文档和实现之间缺少统一同步机制

- 优先级: `P1`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 模板 / 规则层
- 现象:
  - 版本号靠人工记忆同步, README 和说明文档容易滞后.
- 根因:
  - 缺少统一版本真源和一致性检查.
- 风险:
  - 多份文档都像真的, 但不知道哪份是现行标准.
- 本版处理:
  - 只做统一版本真源、统一引用、一致性检查.
  - 不做版本依赖图, 不做自动版本 bump.

### P-005 验证链路不完整, mock 与真机路径经常分叉

- 优先级: `P0`
- 影响层级:
  - 生成器代码层
  - 文档层
  - 模板 / 规则层
- 现象:
  - mock、协议验证、真机验收之间没有稳定闭环.
- 根因:
  - 缺少分阶段验收模板和统一校验入口.
- 风险:
  - 本地看起来可用, 真机一跑就失真.
- 本版处理:
  - 本版纳入 `P0`, 需要以 `M0 / M1 / M2 / M3` 模板和 `validate` 最小校验形成闭环.

### P-006 编码、脚本和环境兼容性治理不足

- 优先级: `P1`
- 影响层级:
  - 文档层
  - 模板 / 规则层
- 现象:
  - PowerShell 中文、UTF-8、pytest 临时文件等问题持续干扰实施.
- 根因:
  - 缺少稳定的环境规则和最小护栏.
- 风险:
  - 大量非业务问题消耗联调时间.
- 本版处理:
  - 只做规则文档和 `validate` 最小护栏.
  - 不做 Git Hook 模板和自动安装.

### P-007 经验记录很多, 但缺少从补丁记录回收成前置能力的机制

- 优先级: `P0`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 模板 / 规则层
- 现象:
  - 经验已经在原始记录和外部 progress 中存在, 但没有稳定回流成规则.
- 根因:
  - 经验沉淀和模板化链路不完整.
- 风险:
  - 同类问题反复重来.
- 本版处理:
  - 本版纳入 `P0`, 重点把已验证经验提升成模板、约束和检查项.

### P-008 协议定义与实现之间缺少一致性保证机制

- 优先级: `P0`
- 影响层级:
  - `.agents` 真源层
  - 文档层
  - 生成器代码层
  - 模板 / 规则层
- 现象:
  - 协议文档、Schema、实现、测试之间经常出现口径分叉.
- 根因:
  - 缺少 Contract-First 的最小一致性约束.
- 风险:
  - 集成阶段才发现协议不兼容.
- 本版处理:
  - 本版纳入 `P0`, 先做存在性和版本一致性检查.
  - 不做深度语义自动比对.

### P-009 根目录与说明文档入口仍然混乱

- 优先级: `P1`
- 影响层级:
  - 文档层
  - 模板 / 规则层
- 现象:
  - 根目录同时混放入口文件、适配输出和治理资料, 新接手者不容易判断阅读顺序.
- 根因:
  - 缺少根目录复杂度控制规则和入口分层说明.
- 风险:
  - 后续升级时文档会继续膨胀.
- 本版处理:
  - 只做目录说明、阅读顺序、入口收敛.
  - 不做目录迁移.

### P-010 提交消息生成与命名规律缺少稳定约束

- 优先级: `P1`
- 影响层级:
  - 文档层
  - `.agents` 真源层
  - 模板 / 规则层
- 现象:
  - 提交消息和版本语义容易偏离既有命名风格.
  - 版本号可能被工具或流程误带成更高版本.
- 根因:
  - 缺少命名样式样本、提交消息约束和“默认不自动升版”的固定规则.
- 风险:
  - 提交历史难检索, 文档标题和版本标题继续分叉.
- 本版处理:
  - 只做命名样本、提交消息模板约束、不自动升版护栏.
  - 不做提交消息自动生成器.

## 六、接口约束

### 1. `project_version`

- 机器真源字段名: `project_version`
- 机器真源格式: `major.minor.patch`
- 示例: `1.0.0`
- 展示层文档可写: `v1.0.0`
- 默认不自动升版, 只能人工修改真源后再由规则和校验联动
- 本版不允许留下“两种机器真源格式都可用”的模糊口径

### 2. `init --project-type embedded`

- 本版目标接口为 `init --project-type embedded`
- 该接口负责生成嵌入式项目的默认治理骨架
- 目录规则采用“默认值 + 可覆盖策略”, 不将 `src + lib` 写死成唯一结构

### 3. `validate`

本版 `validate` 至少需要覆盖:

- 嵌入式关键字段存在性
- 必需文档存在性
- `project_version` 格式校验
- 协议文档 / Schema 示例 / 测试计划版本一致性
- `docs/VALIDATION_PLAN.md` 存在性与 `M0 / M1 / M2 / M3` 结构检查
- 中文 UTF-8 最小护栏

## 七、阶段完成判定

### Phase 1

- 完成判定: `P0 / P1` 边界、`project_version` 规则和本版不做项已经固定, 后续不再反复讨论范围。

### Phase 2

- 完成判定: `init --project-type embedded` 能生成完整骨架, 且骨架结构已经冻结为 Phase 3 的校验输入基线。

### Phase 3

- 完成判定: `validate` 能检查本版 `P0` 所需的最小文档、字段和版本一致性, 并识别明显异常。

### Phase 4

- 完成判定: `P1` 最小护栏已经落到规则、模板或说明中, 且没有扩张成完整增强版。

### Phase 5

- 完成判定: 4 条成功标准都有对应验收结果, 并已保留发布收口记录或回退锚点。
