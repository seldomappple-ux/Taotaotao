# 嵌入式项目专属使用指南

这份文档专门讲一个问题:

如果你要把这套治理体系用到 ESP32、STM32 这类嵌入式软硬件项目里, 当前 v1 应该怎么接, 以及哪些能力还只是预留。

## 一、先说清当前状态

当前仓库已经做了两件对嵌入式有用的事:

1. 在 `.agents/profile.yaml` 里预留了 `mcp` 字段组
2. 在规则目录里预留了 `mcp.placeholder` 规则

但当前 v1 **还没有** 这些能力:

- 现成的嵌入式 overlay 规则包
- 自动生成的烧录规则
- 自动生成的串口联调规则
- 可直接拿来用的 MCP 工具契约

所以你要把它接到嵌入式项目时, 正确理解应该是:

- 它现在已经能做“治理骨架”
- 还不能直接替你做“完整嵌入式规则包”

## 二、当前版本最适合怎么用在嵌入式项目里

推荐把它当成:

- 规则治理骨架
- 版本和经验追溯骨架
- 跨账号接手骨架

不推荐把它误用成:

- 已完成的嵌入式自动化框架

## 三、ESP32 / STM32 项目的接入步骤

### 第 1 步: 初始化治理骨架

在你的嵌入式项目根目录里:

```bash
python -m vibe_governance init --target .
```

### 第 2 步: 填项目事实

重点修改:

- `.agents/profile.yaml`

建议最少改这些:

- `project_type`: 改成你自己的值, 例如 `embedded`
- `project_lang`
- `comment_lang`
- `doc_mode`
- `agent_adapters`

### 第 3 步: 先把硬件红线写进 `.agents/RULES.md`

比如:

- 哪些引脚绝对不能乱改
- 哪些供电条件不能违反
- 哪些烧录命令不能自动执行
- 哪些设备需要人工确认后才能操作

注意:

- 这一步先保证“人类可读”
- 当前 v1 还不会把所有这类红线自动变成机器强制规则

### 第 4 步: 把项目硬件结构写进架构决策

去改:

- `.agents/architecture-decisions.yaml`

至少建议固化:

- 主控型号
- 工具链
- 烧录方式
- 调试方式
- 固件与上位机边界
- 上位机与前端边界

### 第 5 步: 为 bring-up 和踩坑建立 `PROGRESS`

嵌入式项目特别适合写 `PROGRESS`, 因为它的坑非常容易跨账号丢失。

建议记录:

- 首次点灯 / 首次联网 / 首次上报成功
- 串口参数踩坑
- 供电问题
- 驱动版本问题
- 固件与上位机协议调整
- 传感器接线或时序问题

## 四、当前版本里 `mcp` 字段到底怎么用

`.agents/profile.yaml` 里现在有:

- `mcp.enabled`
- `mcp.config_path`
- `mcp.allowed_tools`

当前真实作用是:

- 保存你的嵌入式项目是否准备接 MCP
- 给后续扩展留位置
- 在开启时让规则层出现“这里只是占位, 还没实现完整契约”的提示

当前**不**代表:

- 只要填了就能自动联动硬件
- 只要填了就能自动生成烧录脚本

## 五、嵌入式项目的版本迭代怎么做

嵌入式项目建议比纯软件更重视版本记录。

### 每个版本最好都写清这些东西

- 固件版本改了什么
- 上位机有没有跟着改
- 协议有没有不兼容变更
- 烧录方式有没有变化
- 调试工具有没有变化
- 对应的 `PROGRESS` 条目有哪些

### 推荐的最小闭环

1. 在 `.agents/progress/entries/*` 记录联调和踩坑
2. 在 [CHANGELOG.md](./CHANGELOG.md) 记录版本级变化
3. 在 [ARCHITECTURE.md](./ARCHITECTURE.md) 或架构决策里写清边界变化
4. 在 [CONTEXT_MIGRATION.md](./CONTEXT_MIGRATION.md) 里补充新账号接手需要的环境差异

## 六、跨账号同步时, 嵌入式项目要额外同步什么

除了通用治理文件外, 嵌入式项目额外要特别注意这些东西:

- 板卡型号说明
- 接线说明
- 烧录步骤
- 调试串口参数
- 本地工具链版本
- 协议文档
- 关键硬件红线

如果这些东西没写进仓库, 新账号就算拿到了代码, 也很难真正接手项目。

## 七、当前版本最推荐的嵌入式文档落点

### 适合写在根目录说明文档里的

- 项目怎么接手
- 固件 / 上位机 / 前端之间的关系
- 版本迭代说明

### 适合写在 `.agents/RULES.md` 里的

- 项目特有红线
- 人工确认要求
- 设备操作禁令

### 适合写在 `PROGRESS` 里的

- bring-up 过程
- 调试过程
- 踩坑记录
- 经验回流候选

## 八、如果你真的要把嵌入式能力做成 v2, 建议怎么推进

不要直接在当前模板里塞大量分支判断。

更稳的做法是:

1. 先把嵌入式项目里的通用红线沉淀成文档
2. 再从文档里提炼通用 `rule_id`
3. 再决定是否引入真正的 `embedded` overlay
4. 最后才做 MCP 工具契约和自动生成逻辑

## 九、给嵌入式维护者的一句提醒

当前 v1 已经能帮你解决:

- 规则不落地
- 历史丢失
- 换账号难接手
- 多 IDE 规则不一致

但它还没有替你解决:

- 真正的硬件自动化
- 真正的 MCP 硬件联动
- 完整嵌入式安全规则包

所以正确用法是:

- 先用它把治理骨架搭稳
- 再逐步把嵌入式经验升级成下一版能力
